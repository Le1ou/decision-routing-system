# Передача бекендеру: правки после ревью UI и логики

## Источник контракта

- Рабочий контракт для новых и измененных API: `docs/openapi.yaml`.
- Живой Swagger бекенда генерируется из `apps/backend/src/main.py` и должен быть синхронизирован с `docs/openapi.yaml` при реализации.
- Дублирующий YAML-контракт `docs/dss-api.yaml` удален, чтобы не поддерживать два расходящихся файла.

## Что нужно учесть на бекенде

- Добавить роль `top-manager`.
- Архив не является статусом. Нужен отдельный признак или дата, например `archivedAt`.
- Заявки с `archivedAt` не должны попадать в основной список заявок.
- В архив можно отправить заявки в статусах `rejected` и `completed`; статус при этом не меняется.
- Отклоненные заявки (`rejected`) скрываются из основного UI через 7 дней.
- Добавить действия заявки:
  - `cancel` - отмена заявки в статусе `new`; после отмены заявка получает статус `rejected`;
  - `archive` - скрытие заявки из основного UI без смены статуса.
- Автор и руководитель/топ-руководитель могут отменить заявку в статусе `new`.
- `executorComment`, `managerComment` и `resultText` должны быть отдельными полями.
- `resultText` в UI называется "Результат работы" и показывается только для завершенных заявок.
- В карточке/списке заявок нужно возвращать сложность: `assignedComplexity` или `workType.complexity`.
- Для делегированных заявок нужно возвращать `delegation.delegatedByEmployeeId`, чтобы UI показывал, кто делегировал заявку.
- При добавлении сотрудника frontend передает роль `role`. Должность сотрудника не передается из UI, она приходит из AD.
- Нужно удаление сотрудника из системы, не из AD: `DELETE /employees/{employeeId}`.
- Нужно обновление настройки отдела для подтверждения делегирования внутри отдела: `PATCH /departments/{departmentId}/delegation-settings`.
- Связь для справочника видов работ: `вид работы -> сложность -> позиции`.
- Для видов работ нужны `complexity` и `allowedGradeIds`; `allowedGradeIds` ссылаются на позиции. В контракте и коде оставляем термин `grade`, в русском UI показываем "позиция".
- Нужен справочник позиций: `GET /grades`.
- Нужно обновление вида работ: `PATCH /work-types/{workTypeId}`.
- Настройки приоритетов:
  - `department`: коэффициенты по отделам, ключ - `departmentId`;
  - `managerAuthor`: коэффициенты руководителя-автора по отделам, ключ - `departmentId`;
  - `deadline`: общий коэффициент срока исполнения.
- Формула расчета: `коэффициент отдела * коэффициент срока исполнения + коэффициент руководителя-автора`.
- Фактор "Должность автора" убирается из расчета приоритета. Остается фактор "Руководитель как автор".
- Общее правило доступа по отделам: `manager` видит и редактирует только свой отдел, `top-manager` видит и редактирует все отделы.
