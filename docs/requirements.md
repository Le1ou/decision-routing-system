# Требования к системе маршрутизации заявок

Документ описывает актуальные функциональные и нефункциональные требования к системе.
Модель видов работ вынесена в `docs/type-of-work.md`, роли и права пользователей - в
`docs/roles-and-permissions.md`, статусная модель заявки - в `docs/status-model.md`.

## 1. Назначение системы

1.1. Система предназначена для регистрации, маршрутизации, выполнения и анализа
производственных заявок между подразделениями предприятия.

1.2. Система должна обеспечивать единый жизненный цикл заявки: создание автором,
автоматическое или ручное назначение исполнителя, выполнение, делегирование при
необходимости, закрытие, отклонение и последующее архивирование.

1.3. Система должна снижать ручную нагрузку на руководителей за счет автоматической
маршрутизации заявок по приоритету, отделу, виду работ, должности и грейду исполнителя.

1.4. Система должна сохранять историю ключевых событий по заявке: создание, смены
статусов, назначение исполнителя, начало работы, завершение, возврат, делегирование,
архивирование, уведомления и сообщения в чате.

## 2. Пользователи, роли и доступ

2.1. В системе используются четыре кумулятивные роли: автор, исполнитель, руководитель
отдела и топ-менеджер. Старшая роль включает возможности младших ролей.

2.2. Система должна возвращать текущему пользователю его роль, полный набор
унаследованных ролей и набор прав frontend через `GET /auth/me`.

2.3. Видимость данных должна ограничиваться сервером:
- топ-менеджер видит заявки, сотрудников, отчеты и аналитику по всем отделам;
- руководитель видит заявки своего отдела и заявки, делегированные в его отдел;
- автор видит созданные им заявки;
- исполнитель видит назначенные ему заявки.

2.4. Сервер должен сам вычислять доступные действия над конкретной заявкой и возвращать
их в поле `availableActions`. Frontend не должен самостоятельно принимать решения о
допустимости бизнес-действий.

2.5. Подробная ролевая модель приведена в `docs/roles-and-permissions.md`.

## 3. Аутентификация и сотрудники

3.1. Аутентификация выполняется по HTTP Basic Auth на каждом защищенном запросе.
Сессионные токены в текущей реализации не используются.

3.2. Поддерживаются два режима аутентификации:
- `mock` - локальный каталог пользователей из `apps/backend/config.json`;
- `ad` - проверка пароля через LDAP-bind к Active Directory.

3.3. Пользователь Active Directory должен быть отдельно подключен к системе как
сотрудник. Подключение выполняется через `POST /employees`; человек при этом не
создается в AD, а получает локальную запись участия в системе.

3.4. ФИО, логин, отдел и должность сотрудника берутся из каталога. Роль и признак
активности в системе назначаются руководителем или топ-менеджером.

3.5. У сотрудника есть признак активности `isActive`. Неактивные сотрудники не должны
участвовать в автоматическом распределении заявок.

3.6. Удаление сотрудника из системы выполняется как исключение из участия: запись
сохраняется для истории заявок, сотрудник становится неактивным и помечается удаленным.

## 4. Справочники

4.1. Система должна вести справочник отделов. Для отдела хранятся:
- название;
- коэффициент важности отдела для расчета приоритета;
- настройка подтверждения делегирования внутри отдела;
- задержка перед новым назначением заявки сотруднику;
- порог уведомления о приближении дедлайна.

4.2. Руководитель может менять настройки только своего отдела. Топ-менеджер может
управлять настройками любого отдела.

4.3. Система должна вести справочник должностей. Должность приходит из AD и вручную в
системе не редактируется.

4.4. Система должна вести справочник грейдов. Грейды используются для определения
допуска исполнителя к виду работ.

4.5. Система должна вести справочник видов работ по отделам. Управление видами работ
описано в `docs/type-of-work.md`.

## 5. Заявка

5.1. При создании заявки автор указывает:
- название;
- целевой отдел;
- вид работ;
- срок исполнения;
- описание проблемы.

5.2. Автор заявки, отдел и должность автора должны определяться автоматически по
учетной записи пользователя.

5.3. Новая заявка создается в статусе `new` и получает рассчитанный backend приоритет.
Приоритет и статус не передаются с frontend как пользовательские значения.

5.4. Карточка заявки должна содержать:
- идентификатор, название и описание;
- отдел и вид работ;
- автора, назначенного исполнителя и предыдущего исполнителя при возврате;
- статус и уровень приоритета; непрерывная оценка `priority_score` хранится на backend;
- даты создания, обновления, назначения, начала работы, завершения и дедлайн;
- комментарий исполнителя и комментарий руководителя;
- результат выполненных работ;
- назначенную исполнителем сложность;
- признаки незавершенности, просрочки и архивирования;
- вложения;
- сведения о делегировании;
- список доступных действий для текущего пользователя.

5.5. Описание заявки может редактировать автор в статусе `new`. Руководитель в зоне
видимости также может редактировать описание заявки в статусе `new`.

5.6. Вложения загружаются отдельным multipart-запросом
`POST /applications/{id}/attachments`. Frontend не должен работать с S3 напрямую.

## 6. Приоритет

6.1. Система использует четыре уровня приоритета: `low`, `medium`, `high`, `critical`.

6.2. Backend рассчитывает непрерывную оценку приоритета `priority_score` по формуле:

`P = k_отдела * k_времени + k_руководителя + бонус_срочности`

6.3. В формуле используются:
- `k_отдела` - коэффициент важности отдела автора;
- `k_времени` - доля прошедшего времени от создания до дедлайна с учетом веса срока;
- `k_руководителя` - надбавка, если автор заявки является руководителем или
  топ-менеджером;
- `бонус_срочности` - надбавка для заявок, изначально созданных с коротким сроком
  исполнения.

6.4. Итоговая оценка ограничивается диапазоном `[0..1]` и переводится в дискретный
уровень приоритета. Пороги уровней задаются backend.

6.5. Коэффициенты отдела, надбавки руководителя и вес дедлайна настраиваются через
подсистему приоритета. Параметры бонуса срочности берутся из конфигурации backend.

6.6. Приоритет рассчитывается при создании заявки и периодически пересчитывается для
открытых заявок, так как временной фактор меняется по мере приближения дедлайна.

6.7. Должность автора и вид работ напрямую не входят в формулу приоритета. Вид работ
влияет на подбор исполнителя через матрицы допуска.

## 7. Автоматическая маршрутизация

7.1. Система должна автоматически обрабатывать заявки в статусе `new` по убыванию
`priority_score`.

7.2. Автоматическое назначение выполняется внутри отдела заявки. Исполнитель считается
подходящим, если:
- активен;
- имеет роль исполнителя;
- относится к отделу заявки;
- его грейд входит в допустимые грейды вида работ;
- его должность входит в допустимые должности вида работ либо у вида работ нет
  ограничения по должности;
- у него нет активной заявки в статусах `assigned` или `inProgress`;
- у него нет внутреннего делегирования, ожидающего решения руководителя;
- прошла задержка `empl_appl_delay` после завершения предыдущей заявки.

7.3. Если найдено несколько свободных подходящих исполнителей, система назначает
минимально способного: исполнителя с наименьшим допустимым грейдом, а при равенстве -
дольше всех простаивавшего.

7.4. На одном исполнителе одновременно может быть только одна активная заявка.

7.5. Обычная заявка при отсутствии свободного подходящего исполнителя остается в
статусе `new`; руководителю отдела отправляется уведомление о невозможности
автоматического назначения.

7.6. Критичная заявка при отсутствии свободного подходящего исполнителя может вытеснить
у подходящего исполнителя активную некритичную заявку с наименьшим приоритетом.
Вытесненная заявка возвращается в `new`, получает признак незавершенности и ссылку на
предыдущего исполнителя.

7.7. Критичная активная заявка не вытесняется автоматически. Если вытеснение невозможно,
критичная заявка остается в `new`, а руководителю отправляется уведомление о
необходимости ручного назначения.

7.8. Кулдаун назначения соблюдается для всех заявок, включая критичные.

## 8. Ручное управление заявкой

8.1. Руководитель в своей зоне видимости может вручную назначить исполнителя из статусов
`new`, `assigned`, `inProgress` и `delegated`.

8.2. Ручное назначение не проверяет матрицы грейдов и должностей. Это осознанный режим
ручного управленческого решения.

8.3. При ручном назначении нового исполнителя все другие активные заявки выбранного
исполнителя возвращаются в `new` как незавершенные.

8.4. Руководитель может вернуть заявку из `assigned` или `inProgress` в `new` для
перераспределения.

8.5. Исполнитель может начать работу по назначенной заявке, завершить заявку с
результатом работ, отказаться от заявки с возвратом в `new`, а также инициировать
делегирование.

8.6. Автор или руководитель могут отменить заявку в статусе `new`; такая заявка
переходит в `rejected`.

8.7. Завершенные и отклоненные заявки могут быть заархивированы. Архивирование скрывает
заявку из основного списка, но не меняет ее статус.

## 9. Делегирование

9.1. Внешнее делегирование переводит заявку в статус `delegated` и направляет ее на
решение руководителю принимающего отдела.

9.2. Руководитель может инициировать внешнее делегирование из статусов `new` и
`assigned`. Назначенный исполнитель может инициировать внешнее делегирование из статуса
`assigned`.

9.3. При подтверждении внешнего делегирования заявка переходит в `new`, меняет отдел на
принимающий и попадает в очередь маршрутизации нового отдела.

9.4. При отклонении внешнего делегирования заявка возвращается в `new` в исходном
отделе.

9.5. Внутреннее делегирование выполняется исполнителем в пределах своего отдела из
статусов `assigned` и `inProgress`. Исполнитель указывает новую сложность, не ниже
текущей, и при необходимости меняет вид работ.

9.6. Если отдел требует подтверждения внутреннего делегирования, заявка переходит в
`delegated` до решения руководителя. При подтверждении она возвращается в `new` для
перераспределения; при отклонении возвращается исполнителю в `assigned` или
`inProgress`.

9.7. Если подтверждение внутреннего делегирования не требуется, заявка сразу
возвращается в `new` как незавершенная.

## 10. Уведомления

10.1. Система должна создавать внутренние уведомления и отдавать их через
`GET /notifications` с признаком прочтения.

10.2. Уведомления создаются при назначении исполнителя, делегировании, подтверждении или
отклонении делегирования, завершении, отклонении, возврате заявки, вытеснении критичной
заявкой и невозможности автоматического распределения.

10.3. Фоновая подсистема событий должна уведомлять руководителя отдела и назначенного
исполнителя о приближении дедлайна. Порог задается настройкой отдела
`deadline_notification`.

10.4. При наступлении дедлайна открытая заявка помечается как просроченная, а
руководителю отдела и назначенному исполнителю отправляется уведомление.

10.5. Уведомления о приближении дедлайна, просрочке и невозможности распределения
должны дедуплицироваться, чтобы фоновые циклы не создавали повторные сообщения.

## 11. Чат заявки

11.1. В карточке заявки должен быть доступен чат между участниками заявки: автором,
назначенным исполнителем и руководителем в зоне видимости.

11.2. Читать чат может пользователь, которому доступна карточка заявки.

11.3. Писать в чат могут автор, назначенный исполнитель и руководитель в зоне видимости.
После завершения или отклонения заявки чат доступен только на чтение.

11.4. Система должна хранить маркер прочтения чата на пользователя и считать количество
непрочитанных сообщений.

11.5. Новое сообщение должно создавать уведомление адресату. Повторные уведомления не
дублируются, пока адресат не прочитал чат.

## 12. Отчетность и аналитика

12.1. Система должна предоставлять отчет по заявкам в формате JSON для предпросмотра и
выгрузку в файл `applications.xlsx` через совместимый путь
`GET /reports/applications.xls`.

12.2. Отчет по заявкам должен поддерживать фильтры по датам создания и завершения,
статусу и исполнителю.

12.3. Руководитель должен получать отчеты только по своему отделу. Топ-менеджер должен
получать отчеты по всем отделам.

12.4. Система должна предоставлять аналитические срезы по заявкам, исполнителям, видам
работ и отделам через `/analytics/applications`, `/analytics/executors`,
`/analytics/work-types`, `/analytics/departments`.

12.5. Аналитика должна учитывать область видимости пользователя: руководитель - свой
отдел, топ-менеджер - все отделы.

## 13. Вложения и хранение файлов

13.1. Вложения заявок хранятся в S3-совместимом хранилище. В локальном контуре
используется MinIO.

13.2. Backend сохраняет метаданные вложений в БД и возвращает временные ссылки для
скачивания или просмотра.

13.3. Загружать вложения могут только пользователи, которым доступна заявка: автор,
назначенный исполнитель, руководитель в зоне видимости или топ-менеджер.

## 14. Нефункциональные требования

14.1. Все операции изменения состояния заявки должны выполняться сервером транзакционно:
смена статуса, обновление полей, запись истории и изменение связей должны быть
согласованы.

14.2. Доступ к данным должен проверяться на backend для каждого запроса. Нельзя
полагаться только на скрытие элементов интерфейса.

14.3. Запросы к БД должны выполняться параметризованно.

14.4. Фоновые подсистемы событий и маршрутизации должны быть идемпотентными: повторный
тик не должен создавать некорректные дубли уведомлений или нарушать инвариант одной
активной заявки на исполнителя.

14.5. Автоматическая маршрутизация должна сериализовать проходы распределения, чтобы два
параллельных прохода не назначили одного исполнителя на разные заявки.

14.6. Система должна сохранять состояние подключения сотрудников к системе и поддерживать
резервное копирование/восстановление данных в релизном контуре.

14.7. Время в backend должно храниться и обрабатываться в формате с часовым поясом.

14.8. API должно отдавать стабильные DTO для frontend: строки для идентификаторов,
ISO-строки для дат и явные списки вложенных сущностей.

14.9. Система должна поддерживать постраничную выдачу списка заявок и фильтрацию по
статусу, приоритету, авторству, назначению, делегированию, исполнителю и идентификатору.

14.10. Справочники и действия управления должны быть ограничены областью ответственности
руководителя; топ-менеджер является исключением с доступом ко всем отделам.
