# Интеграция с Active Directory

Документ описывает, как переключить аутентификацию backend с режима mock
(по умолчанию) на реальный Active Directory (AD) по протоколу LDAP.

## Краткая суть

- По умолчанию backend работает в режиме **mock**: пароли проверяются по
  справочнику `MOCK_AD` из `apps/backend/config.json`. Этот режим не требует
  AD-сервера и используется для локальной разработки и тестов.
- В режиме **ad** backend проверяет пароль пользователя через LDAP-bind к
  контроллеру домена. Всё остальное (роль, отдел, `employee_id`) по-прежнему
  берётся из системного справочника — **AD отвечает только за пароль**.
- Переключение между режимами — это изменение одного параметра `AUTH_MODE`.
  Код менять не нужно.

## Переключение на AD за два шага

1. Установите режим AD:
   - в `.env`: `AUTH_MODE=ad`
   - (либо в `apps/backend/config.json`: `"AUTH_MODE": "ad"`)
2. Укажите параметры подключения к AD — через переменные окружения в `.env`
   **или** в блоке `AD_auth` файла `apps/backend/config.json`.

После этого перезапустите backend (`make up` / пересборка контейнера).

## Параметры подключения

Переменные окружения имеют приоритет над `config.json` — это позволяет задавать
адрес сервера на этапе развёртывания, не редактируя файлы в репозитории.

| Переменная окружения | Поле в `config.json` → `AD_auth` | Назначение | По умолчанию |
|----------------------|----------------------------------|------------|--------------|
| `AD_SERVER`      | `server_adress`  | Имя хоста или IP контроллера домена | — (обязательно) |
| `AD_DOMAIN`      | `Domain`         | Домен AD для UPN вида `user@DOMAIN`  | — (обязательно) |
| `AD_USE_SSL`     | `use_ssl`        | Использовать LDAPS (TLS)             | `true` |
| `AD_PORT`        | `port`           | Порт LDAP                            | `636` (SSL) / `389` (без SSL) |
| `AD_ROLE_SOURCE` | `role_source`    | Откуда брать роль: `system` или `ad` | `system` |
| `AD_SEARCH_BASE` | `search_base`    | База поиска LDAP (для режима ролей из AD) | вычисляется из домена |
| —                | `group_role_map` | Карта «группа AD → роль системы» (только в `config.json`) | `{}` |

`AUTH_MODE` задаётся переменной `AUTH_MODE` (env) или полем `AUTH_MODE` в
`config.json`. Значения: `mock` (по умолчанию) или `ad`.

### Пример: `.env`

```dotenv
AUTH_MODE=ad
AD_SERVER=dc01.company.local
AD_DOMAIN=company.local
AD_USE_SSL=true
AD_PORT=636
```

### Пример: `apps/backend/config.json`

```json
{
  "AUTH_MODE": "ad",
  "AD_auth": {
    "server_adress": "dc01.company.local",
    "Domain": "company.local",
    "use_ssl": true,
    "port": 636
  }
}
```

`docker-compose.local.yml` уже пробрасывает `AUTH_MODE`, `AD_SERVER`,
`AD_DOMAIN`, `AD_USE_SSL`, `AD_PORT` в контейнер backend.

## Как работает аутентификация в режиме AD

При каждом защищённом запросе (Basic Auth) backend:

1. Берёт логин и пароль из заголовка `Authorization`.
2. Выполняет LDAP-bind к `AD_SERVER` под учётной записью `логин@AD_DOMAIN`
   (SIMPLE bind). Если bind не прошёл — ответ **401**.
3. Проверяет, что пользователь **заведён в системе** (есть запись `inSystem` в
   справочнике). Если нет — ответ **403** (`User is not onboarded into the system`).
4. При успехе запрос обрабатывается; роль, отдел и `employee_id` берутся из
   системного справочника, как и в режиме mock.

> Важно: AD проверяет **только пароль**. Роль сотрудника в системе AD не хранится
> и из AD не вычитывается — она назначается при заведении сотрудника (см. ниже).

## Заведение пользователей (onboarding)

Чтобы AD-пользователь мог работать в системе, он должен быть заведён как
участник: у него должна появиться запись сотрудника (роль, отдел, `employee_id`).
Это делается через `POST /employees` (экран «Управление сотрудниками»):
руководитель выбирает пользователя из AD и назначает ему роль. Должность
приходит из AD, из системы пользователь не удаляется в AD при удалении.

Пользователь, прошедший проверку пароля в AD, но не заведённый в системе,
получит **403**.

## Роль из групп AD (опционально)

По умолчанию (`role_source = system`) роль пользователя берётся из системного
справочника — той, что назначена при заведении сотрудника. Опционально роль
можно вычислять из членства пользователя в группах AD.

Включение:

1. Задайте `AD_ROLE_SOURCE=ad` (env) или `"role_source": "ad"` в `AD_auth`.
2. Опишите соответствие групп ролям в `AD_auth.group_role_map` в
   `apps/backend/config.json`.
3. При необходимости задайте `AD_SEARCH_BASE` / `search_base` — базу поиска LDAP.
   Если не задана, она вычисляется из домена (`company.local` → `DC=company,DC=local`).

Пример `apps/backend/config.json`:

```json
{
  "AUTH_MODE": "ad",
  "AD_auth": {
    "server_adress": "dc01.company.local",
    "Domain": "company.local",
    "use_ssl": true,
    "port": 636,
    "role_source": "ad",
    "search_base": "DC=company,DC=local",
    "group_role_map": {
      "CN=DRS-TopManagers": "top-manager",
      "CN=DRS-Managers": "manager",
      "CN=DRS-Executors": "executor"
    }
  }
}
```

Как это работает:

- После успешного bind backend читает атрибут `memberOf` пользователя
  (поиск по `userPrincipalName` или `sAMAccountName`).
- Каждый ключ из `group_role_map` ищется как подстрока в DN групп пользователя
  (без учёта регистра) — можно указывать как полный DN, так и просто `CN=...`.
- Если совпало несколько групп, выбирается **наивысшая** роль по лестнице ролей
  `author → executor → manager → top-manager`.
- Если не совпала ни одна группа, остаётся роль из справочника (понижения не
  происходит).

Важные замечания:

- Роль из AD вычисляется **при каждом входе** и переопределяет роль в справочнике
  для этого пользователя.
- Пользователь по-прежнему должен быть заведён в системе (`inSystem`) — из AD
  берётся только роль, а `employee_id`, отдел и прочее остаются из справочника.
- Назначение прав в PostgreSQL фиксируется при первом подключении пользователя в
  рамках процесса. Смена роли в AD полностью применяется после «холодного»
  переподключения (перезапуск backend), как и смена пароля (см. ниже).

## Безопасность

- **Используйте LDAPS (`AD_USE_SSL=true`, порт 636).** При обычном LDAP пароль
  передаётся в открытом виде. По умолчанию SSL включён.
- Пустой пароль отклоняется до обращения к AD (защита от «unauthenticated bind»,
  который в LDAP формально проходит без проверки пароля).
- Таймаут подключения к AD — `AD_CONNECT_TIMEOUT` (10 секунд): недоступный или
  неверно указанный сервер приводит к быстрому отказу, а не к зависанию запроса.

## Пароли ролей PostgreSQL и смена пароля в AD

Для каждого пользователя backend создаёт отдельную роль в PostgreSQL и
подключается к БД под ней, используя введённый (проверенный в AD) пароль. При
повторном входе пароль роли PostgreSQL синхронизируется с текущим.

Следствие: если пользователь сменил пароль в AD, новый пароль начнёт
использоваться для роли PostgreSQL при следующем «холодном» подключении
(после перезапуска backend пул соединений пересоздаётся). В рамках уже
поднятого пула старые соединения продолжают работать.

## Диагностика

| Симптом | Причина | Что проверить |
|---------|---------|---------------|
| Все входы возвращают **401** | Неверные учётные данные или недоступен AD | Доступность `AD_SERVER:AD_PORT`, корректность `AD_DOMAIN`, логи backend (`[auth] AD bind error ...`) |
| Вход возвращает **403** | Пользователь не заведён в системе | Завести через `POST /employees` (должна появиться запись `inSystem`) |
| Вход возвращает **500** `Active Directory is not configured` | В режиме `ad` не задан сервер или домен | Заполнить `AD_SERVER`/`AD_DOMAIN` (или `AD_auth` в `config.json`) |
| В логах при старте `WARNING: AUTH_MODE=ad but the AD server/domain are not configured` | Режим AD включён, но настройки не заданы | То же, что выше |
| Запросы к AD «зависают» / долго отвечают | Сетевой доступ к контроллеру домена | Firewall/маршрутизация до `AD_SERVER:AD_PORT`; таймаут — 10 секунд |

При старте в режиме AD backend печатает строку вида
`[auth] Active Directory mode enabled (server=...:636, ssl=True).` — по ней
удобно убедиться, что режим и параметры применились.

## Что НЕ входит в текущую интеграцию

- **Автоматическая синхронизация справочника** сотрудников из AD. Участник всё
  равно заводится через `POST /employees` (создаётся запись с `employee_id`,
  отделом и т.д.); из AD берутся только пароль и — опционально — роль.
- **Маппинг ролей из вложенных групп.** Учитывается прямое членство пользователя
  (атрибут `memberOf`); рекурсивный обход вложенных групп не выполняется.

## Возврат к режиму mock

Установите `AUTH_MODE=mock` (или уберите переменную — это значение по умолчанию)
и перезапустите backend. Режим mock работает без AD-сервера и используется в
тестах.
