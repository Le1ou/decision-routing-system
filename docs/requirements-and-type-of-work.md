# Бизнес-требования к системе

## Функциональные требования

### 1. Общие требования к системе
1.1. Реализовать систему для работы с производственными заявками.
1.2. Все созданные заявки попадают в статус «Новый».
1.3. В систему заводятся все сотрудники организации.
1.4. У каждого сотрудника должен быть статус «Активен» / «Неактивен» (настраивается руководителем). Заявки могут приходить только активным сотрудникам.

### 2. Ролевая модель и справочники
2.1. В системе должны быть следующие роли:
• Автор (создатель заявки)
• Исполнитель (кто выполняет заявку)
• Руководитель исполнителей отдела (переназначает заявки между исполнителями, определяет виды работ для своего отдела)
2.2. Реализовать справочник исполнителей по отделам. Для каждого отдела заводится руководитель.
2.3. Реализовать справочник должностей с указанием приоритетности должности.
2.4. Реализовать справочник отделов с указанием приоритетности отдела.
2.5. Реализовать возможность создания словаря видов работ по отделу. Словарь формирует руководитель отдела.
2.6. Реализовать зависимость вида работы из словаря с должностью исполнителя.

### 3. Структура и атрибуты заявки
3.1. Заявка должна содержать:
• Дата и время создания заявки
• Дата и время завершения заявки
• Автор (должность и отдел автора)
• Исполнитель (с указанием его отдела; должность подтягивается автоматически при назначении исполнителя)
• Статус заявки
• Время на взятие в работу (фактическое)
• Время на обработку (фактическое, запускается после взятия в работу)
• Поле с описанием проблемы (заполняется на этапе создания, можно редактировать)
• Фото (несколько файлов)
• Комментарии от исполнителя
• Вид работ (выбор из справочника)
• Состав / отчет работ от исполнителя
• Приоритет заявки
3.2. Поля для заполнения при создании заявки:
• Автор (автоматически)
• Вид работ (два справочника: сначала департамент/отдел, затем вид работ)
• Тема / название / заголовок заявки
• Описание проблемы + необходимые файлы (фото)
• Должность автора (автоматически из справочника)
• Отдел автора (автоматически из справочника)

### 4. Статусы заявки
4.1. Статусы заявки:
• Новый / создана (бэклог)
• Назначен исполнитель (ещё не взял в работу)
• Делегирована в другой отдел
• В работе (работа идет)
• Отклонена (дубликат)
• Завершена

### 5. Приоритет заявки
5.1. Статусы приоритета:
• Низкий
• Средний
• Высокий
• Критичный
5.2. Приоритет формируется на основе:
• Отдела автора
• Должности автора
• Вида работ
• Даты, к которой необходимо выполнить заявку
• Приоритета от руководителя, если он является автором заявки
5.3. Руководитель может менять приоритет заявки, когда она находится в статусе «Новый», в течение времени, указанного в п. 11.

### 6. Правила распределения и перераспределения заявок
6.1. Реализовать алгоритм распределения и перераспределения заявок.
6.2. Заявки из статуса «Новый» распределяются в первую очередь более приоритетные.
6.3. На одном исполнителе может находиться только одна заявка в любом статусе («В работе» / «Назначен исполнитель»).
6.4. Автоматическое перераспределение заявок происходит только в случае возникновения критических заявок.
6.5. При перераспределении из-за новой критической заявки:
• Заявка возвращается в статус «Новый» из статусов «В работе» или «Назначен исполнитель»
• Заявке присваивается признак «Незавершенная»
6.6. Заявку в статусе «Критичный» нельзя автоматически перераспределять с исполнителя. Такая заявка перераспределяется только руководителем вручную.

### 7. Делегирование заявок
7.1. Реализовать делегирование заявок из одного департамента в другой с подтверждением руководителя принимающей стороны.
7.2. Делегирование в рамках своего отдела возможно до статуса «В работе» (включительно).
7.3. Делегирование в другой отдел возможно только в статусе «Назначен исполнитель».
7.4. Делегированной заявке всегда присваивается статус «Новый».
7.5. Делегирование возможно для заявок с любым приоритетом.
7.6. Каждый руководитель устанавливает для своего департамента настраиваемый параметр: требуется ли его подтверждение при делегировании в рамках своего отдела.

### 8. Функционал руководителя
8.1. Руководитель может:
• Отправить заявку в другой отдел из статуса «Назначен исполнитель»
• Вручную назначить исполнителя заявки из статуса «Новый»
• Перевести заявку из статусов «В работе» или «Назначен исполнитель» в статус «Новый» для дальнейшего перераспределения (автоматически или вручную)
• Вручную установить тип заявки, даже если он не соответствует матрице грейдов
• Настраивать статус «Активен» / «Неактивен» для сотрудников своего отдела

### 9. Сложность заявок и грейды
9.1. У заявок будет сложность, определяемая видом работ. Сложность влияет на то, какой грейд сотрудника может выполнять заявку.
9.2. При переадресации заявки в свой же отдел сотрудник выбирает параметр сложности на свой взгляд. Нельзя поставить сложность ниже исходной.
9.3. Параметр сложности не отображается у автора заявки.

### 10. Учет рабочего времени и квоты
10.1. Реализовать временной период (настраиваемый параметр), через сколько времени сотруднику назначится новая заявка после завершения предыдущей.
10.2. Необходимо фиксировать рабочее время сотрудников в системе.

### 11. Система уведомлений
11.1. Реализовать внутреннюю систему уведомлений по заявкам.
11.2. Система должна уведомлять руководителя о заявке в статусе «Новый», если осталось 25% времени от момента создания до предполагаемого времени окончания (настраиваемый параметр).

### 12. Система отчетности
12.1. Реализовать систему отчетности для просмотра выполнения работ исполнителей.

### 13. Дополнительные правила
13.1. Переадресация в рамках одного отдела происходит до взятия в работу (статус «В работе» включительно). Исполнитель меняет вид работ, заявка возвращается в статус «Новый».

## Отделы и виды работ на промышленном предприятии

1. **IT-отдел**
• Замена оборудования
• Починка оборудования
• Настройка сервера
• Установка ПО
• Настройка ПО

2. **Отдел главного энергетика (ОГЭ)**
• Ремонт электродвигателей
• Устранение замыканий
• Ремонт проводки
• Ремонт трубопроводов
• Устранение обесточивания

3. **Производственный отдел**
• Заявка на ремонт оборудования
• Заявка на замену оборудования

4. **Отдел контроля качества (ОКК)**
• Проверка/приемка покупных деталей
• Проверка/приемка готовой продукции
• Фиксация брака
• Устранение брака

5. **Отдел главного механика (ОГМ)**
• Ремонтные работы
• Замена комплектующих
• Замена масла
• Аварийные работы

6. **Складской отдел**
• Оформление товарных накладных
• Отгрузка товара
• Инвентаризация
• Приемка товара
• Списание и утилизация
• Приемка от производства (внутри организации)

7. **Отдел снабжения**
• Предоставление комплектующих/деталей
• Оформление заказа поставщику
• Создание заявок на тендер
• Подготовка договоров
• Замена позиции в поставке
