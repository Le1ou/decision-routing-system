# Бизнес-требования к системе

> Документ приведён в соответствие с фактической реализацией backend. Где поведение
> кода отличается от исходного требования — текст исправлен; где есть нюансы
> реализации — добавлены пометки *Реализация:*; ещё не реализованные правила
> помечены *(в разработке)*. Перечень внесённых правок — в `docs/doc-updates.md`.

## Функциональные требования

### 1. Общие требования к системе
1.1. Система предназначена для работы с производственными заявками.
1.2. Все созданные заявки попадают в статус «Новый».
1.3. Сотрудники заводятся в систему из каталога пользователей (Active Directory; в
локальном/тестовом режиме — каталог `MOCK_AD` в `apps/backend/config.json`). Заведение
участника выполняется через `POST /employees` (экран «Управление сотрудниками»); из AD
берутся ФИО, отдел и должность, роль назначает руководитель. См. `apps/backend/docs/ad-integration.md`.
1.4. У каждого сотрудника есть признак «Активен» / «Неактивен» (поле `is_active`,
меняется через `PATCH /employees/{id}` руководителем своего отдела). *Реализация:*
учёт активности при автоматическом назначении — *(в разработке)*, см. §6.

### 2. Ролевая модель и справочники
2.1. В системе четыре **кумулятивных** роли (каждая включает права предыдущей):
• Автор (создатель заявки)
• Исполнитель (выполняет заявку; включает права автора)
• Руководитель отдела (управляет заявками своего отдела, сотрудниками, видами работ;
включает права исполнителя и автора)
• Топ-менеджер (то же по всем отделам; включает все нижестоящие роли)
*Реализация:* у сотрудника в справочнике хранится одна роль; текущему пользователю
(`GET /auth/me`) возвращается полный набор ролей и набор прав (`permissions`). Различие
руководитель/топ-менеджер — это **область видимости** (свой отдел против всех отделов).
2.2. Справочник сотрудников по отделам. Видимость и управление ограничены отделом для
руководителя; топ-менеджер видит все отделы.
2.3. Справочник должностей (`positions`, таблица `post`). Должность приходит из AD,
вручную не редактируется, используется для отображения. *Реализация:* отдельной
«приоритетности должности» в расчёте приоритета **нет** (см. §5).
2.4. Справочник отделов. У отдела есть **коэффициент важности** (`department.value`,
0..1) — используется как коэффициент отдела в формуле приоритета (§5), а также параметры
`empl_appl_delay` и `deadline_notification` (§10, §11), настраиваемые руководителем через
`PATCH /departments/{id}`.
2.5. Словарь видов работ по отделу формирует руководитель отдела
(`POST/PATCH/DELETE /work-types`, в рамках своего отдела). Удаление вида работ, на который
ссылаются заявки, запрещено (`409`).
2.6. Вид работы связан с допустимыми **грейдами** через матрицу
(`type_of_work_to_grade`, поле `allowedGradeIds`). Грейд связан с должностью через
`post_grade`. *Реализация:* автоматический подбор исполнителя по грейду — *(в разработке)*,
см. §6; при ручном назначении матрица не проверяется (см. §8).

### 3. Структура и атрибуты заявки
3.1. Заявка содержит:
• Дата и время создания и обновления (`createdAt`, `updatedAt`)
• Дата и время завершения (`finishedAt`)
• Срок исполнения / дедлайн (`deadlineAt`)
• Автор (отдел и должность автора)
• Исполнитель (отдел и должность подтягиваются при назначении); предыдущий исполнитель
(`previousExecutor`) при возврате заявки
• Статус заявки и приоритет
• Дата назначения исполнителя (`assignedAt`) и взятия в работу (`startedAt`)
• Описание проблемы (редактируется автором в статусе «Новый»)
• Вложения / фото (несколько файлов, хранятся в S3 — см. §15)
• Комментарий исполнителя и комментарий руководителя
• Вид работ (из справочника) и сложность (назначенная — `assignedComplexity`)
• Результат / состав работ исполнителя (`resultText`, заполняется при завершении)
• Признак «Незавершённая» (`isUnfinished`) и признак просрочки (`isExpired`)
3.2. Поля при создании заявки (`POST /applications`): тема/название, отдел, вид работ,
**срок исполнения (дедлайн)**, описание. Автор, его отдел и должность проставляются
автоматически из учётной записи. *Реализация:* файлы при создании не передаются —
вложения загружаются отдельным запросом `POST /applications/{id}/attachments` (§15).

### 4. Статусы заявки
4.1. Статусы заявки (внутренние идентификаторы в скобках):
• Новый / создана (`new`)
• Назначен исполнитель (`assigned`)
• Делегирована в другой отдел (`delegated`)
• В работе (`inProgress`)
• Отклонена (`rejected`)
• Завершена (`completed`)
4.2. Подробная статусная модель и переходы — в `docs/status-model.md`. Дополнительно у
заявки есть признак **архивирования** (`archivedAt`): завершённые и отклонённые заявки
можно убрать из основного списка; отклонённые автоматически скрываются из списка через
7 дней. Архивирование статус не меняет.

### 5. Приоритет заявки
5.1. Уровни приоритета: Низкий (`low`), Средний (`medium`), Высокий (`high`),
Критичный (`critical`).
5.2. Приоритет рассчитывается по формуле:

`П = k_отдела · k_времени + k_руководителя + бонус_срочности`

где:
• **k_отдела** — коэффициент важности отдела автора в диапазоне **[0.5, 1.25]**
(по умолчанию = `department.value`); важные отделы (>1) ближе к дедлайну дают critical;
• **k_времени** — фактор срока: плавная доля прошедшего времени `[0..1]`, умноженная на
настраиваемый вес срока (**по умолчанию 1.0** — время считается «в полную силу»);
• **k_руководителя** — надбавка, применяемая, когда **автор заявки — руководитель**;
её значение (`managerAuthor` по отделу, по умолчанию 0.2) задаёт **топ-менеджер**;
• **бонус_срочности** — фиксированная надбавка, если до дедлайна остаётся не больше
порога (по умолчанию 24 часа → +0.5; параметры в `config.json → priority`).

Итоговое значение нормируется в `[0..1]` (`priority_score`) и отображается одним из
четырёх уровней (§5.1). *Реализация:* «должности автора» и «вида работ» в формуле **нет**
(вид работ влияет только на сложность/грейды, §9). k_отдела/k_руководителя берутся по
отделу автора из настроек приоритета (`/priority-settings`; дефолт k_отдела =
`department.value`, managerAuthor = 0.2, вес срока = 1.0); параметры бонуса срочности — в
`config.json → priority`. Коэффициенты задаёт топ-менеджер (рядовой руководитель видит свой
отдел только на чтение; коэффициент отдела допускается до 1.25). **Расчёт реализован**
(модуль `priority_module`): `priority_score` вычисляется при создании заявки и периодически
пересчитывается подсистемой событий (k_времени растёт со временем). Сама формула вынесена в
отдельную функцию — её можно заменить или добавить варианты, не затрагивая остальной код.
5.3. Надбавка руководителя задаётся не вручную на конкретной заявке, а через коэффициент
`managerAuthor` (см. §5.2) и применяется автоматически, когда автор — руководитель.
*Реализация:* отдельного действия для ручного изменения приоритета конкретной заявки нет.

### 6. Правила распределения и перераспределения заявок *(реализовано)*
Автоматическую маршрутизацию выполняет подсистема `routing_module` (запускается фоновым
циклом отдельным шагом после подсистемы событий). Ручное назначение руководителем (§8)
тоже доступно. Правила:
6.1. Алгоритм автоматического распределения и перераспределения заявок: на каждом проходе
обрабатываются заявки в статусе «Новый» по убыванию `priority_score`.
6.2. Из статуса «Новый» в первую очередь распределяются более приоритетные заявки.
6.3. На одном исполнителе одновременно только одна активная заявка («Назначен исполнитель»
/ «В работе»). *Реализация (готово):* при назначении исполнителя его прежняя активная
заявка автоматически возвращается в «Новый» с признаком «Незавершённая» и сохранением
ссылки на предыдущего исполнителя (см. §8).
6.4. Автоматическое перераспределение происходит только при появлении критических заявок;
обычные заявки автоматически не перераспределяются.
6.5. При вытеснении критической заявкой прежняя заявка возвращается в «Новый» с признаком
«Незавершённая».
6.6. Критическую заявку нельзя перераспределить автоматически — только руководителем
вручную.
6.7. Кулдаун `empl_appl_delay` (§10.1) соблюдают все заявки, включая критические.
6.8. Автоматически назначается минимально подходящий по грейду свободный исполнитель из
нужного отдела; при отсутствии свободных критическая заявка вытесняет наименее
приоритетную; если вытеснять некого — заявка ждёт и руководителю уходит уведомление.

### 7. Делегирование заявок *(реализовано)*
7.1. Делегирование заявки в другой отдел с подтверждением руководителя принимающей
стороны.
7.2. Делегирование в рамках своего отдела (изменение сложности/вида работ исполнителем)
возможно до статуса «В работе» включительно (`delegateInternal`).
7.3. Делегирование в другой отдел (`delegateExternal`) доступно руководителю из статусов
«Новый» и «Назначен исполнитель», а назначенному исполнителю — из «Назначен исполнитель».
7.4. При подтверждении внешнего делегирования заявка переходит в статус «Новый» и
**переходит во владение целевого отдела** (`department_id` меняется на отдел-получатель);
при отклонении остаётся в исходном отделе со статусом «Новый».
7.5. Делегирование возможно для заявок с любым приоритетом.
7.6. Каждый руководитель задаёт для своего отдела параметр `delegated_to_same_dep` —
требуется ли его подтверждение при делегировании в рамках своего отдела
(`PATCH /departments/{id}/delegation-settings`).

### 8. Функционал руководителя *(реализовано)*
8.1. Руководитель (в рамках своего отдела; топ-менеджер — по всем отделам) может:
• Назначить исполнителя заявки (`assignExecutor`) из статусов «Новый», «Назначен
исполнитель», «В работе», «Делегирована» — при этом прежняя активная заявка выбранного
исполнителя освобождается (см. §6.3)
• Отправить заявку в другой отдел (`delegateExternal`) из «Новый» / «Назначен исполнитель»
• Вернуть заявку из «Назначен исполнитель» / «В работе» в «Новый» (`returnToNew`) с
признаком «Незавершённая»
• Отклонить заявку (`reject`) из «Назначен исполнитель» / «В работе»
• Подтвердить или отклонить делегирование (`confirmExternalDelegation` /
`declineExternalDelegation`)
• Изменить вид работ (`changeWorkType`) и отменить заявку (`cancel`) в статусе «Новый»
• Назначить исполнителя независимо от матрицы грейдов (при ручном назначении соответствие
грейда не проверяется)
• Настраивать признак «Активен» / «Неактивен» сотрудников и параметры отдела
8.2. Перечень доступных действий для конкретной заявки и пользователя backend вычисляет
сам и возвращает в карточке (`availableActions`); попытка недоступного действия → `403`.

### 9. Сложность заявок и грейды
9.1. У заявки есть сложность (`easy`/`medium`/`hard`/`critical`), определяемая видом работ;
сложность связана с допустимыми грейдами исполнителя (матрица §2.6).
9.2. При делегировании в свой же отдел (`delegateInternal`) исполнитель задаёт новую
сложность — не ниже текущей (иначе `400`); может также сменить вид работ.
9.3. Параметр сложности не показывается автору заявки (логика отображения на frontend).

### 10. Учёт рабочего времени и квоты
10.1. Параметр `empl_appl_delay` (минуты) — задержка перед назначением новой заявки
сотруднику, недавно завершившему предыдущую. Настраивается руководителем для своего
отдела (`PATCH /departments/{id}`). *Реализация:* учёт задержки при автоназначении —
*(в разработке)*, см. §6.
10.2. Рабочее время и время по этапам жизненного цикла фиксируются журналом переходов
статусов (`application_status_history`) — пишется при каждой смене статуса; используется
аналитикой (§12).

### 11. Система уведомлений *(реализовано)*
11.1. Внутренние уведомления по заявкам (таблица `notification`, `GET /notifications`,
отметка прочитанным). Создаются на событиях: назначение исполнителя, делегирование,
подтверждение/отклонение делегирования, завершение и отклонение заявки.
11.2. Подсистема событий (фоновый цикл) уведомляет **руководителя отдела и назначенного
исполнителя** о приближении дедлайна, когда доля оставшегося времени становится не больше
параметра `deadline_notification` отдела (`[0..1]`, по умолчанию 0.25), по всем открытым
статусам; а также помечает просроченные заявки (`isExpired`) и уведомляет о просрочке.

### 12. Система отчётности и аналитики *(реализовано)*
12.1. Отчёт по выполнению заявок (`GET /reports/applications`) и выгрузка в XLS
(`GET /reports/applications.xls`) с фильтрами по датам/статусу/исполнителю. Рядовой
руководитель видит только свой отдел.
12.2. Аналитика (`GET /analytics/{applications,executors,work-types,departments}`):
статистика по заявкам, исполнителям, видам работ и отделам, с фильтром по периоду и
ограничением по отделу. Контракт — `apps/backend/docs/analytics-contract.md`.

### 13. Дополнительные правила
13.1. Переадресация в рамках одного отдела (`delegateInternal`) возможна до взятия в работу
включительно («В работе»). Исполнитель задаёт сложность (не ниже текущей) и при
необходимости меняет вид работ; заявка возвращается в «Новый» (или сначала в «Делегирована»
на подтверждение руководителя, если у отдела включён `delegated_to_same_dep`).

### 14. Аутентификация и доступ *(реализовано)*
14.1. Аутентификация — HTTP Basic Auth на каждом запросе (токенов/сессий нет). Режимы:
`mock` (пароли из `config.json`) и `ad` (проверка пароля через LDAP-bind к Active
Directory). См. `apps/backend/docs/ad-integration.md`.
14.2. Доступ к разделам управления определяется правами (`permissions`) текущего
пользователя; видимость заявок и справочников ограничивается отделом для руководителя и
не ограничивается для топ-менеджера.

### 15. Хранилище вложений *(реализовано)*
15.1. Вложения (фото) заявок хранятся в S3-совместимом хранилище (локально — MinIO).
Загрузка — `POST /applications/{id}/attachments` (multipart), выдача — временные
presigned-ссылки в карточке заявки. См. `apps/backend/docs/s3-integration.md`.

## Отделы и виды работ на промышленном предприятии

> Соответствует наполнению по умолчанию (seed). Отделы и виды работ далее редактируются
> руководителями (§2.5).

1. **IT-отдел**
• Замена оборудования
• Починка оборудования
• Настройка сервера
• Установка ПО
• Настройка ПО

2. **Отдел главного энергетика (ОГЭ)**
• Ремонт электродвигателей
• Устранение замыканий
• Ремонт проводки
• Ремонт трубопроводов
• Устранение обесточивания

3. **Производственный отдел**
• Заявка на ремонт оборудования
• Заявка на замену оборудования

4. **Отдел контроля качества (ОКК)**
• Проверка/приемка покупных деталей
• Проверка/приемка готовой продукции
• Фиксация брака
• Устранение брака

5. **Отдел главного механика (ОГМ)**
• Ремонтные работы
• Замена комплектующих
• Замена масла
• Аварийные работы

6. **Складской отдел**
• Оформление товарных накладных
• Отгрузка товара
• Инвентаризация
• Приемка товара
• Списание и утилизация
• Приемка от производства (внутри организации)

7. **Отдел снабжения**
• Предоставление комплектующих/деталей
• Оформление заказа поставщику
• Создание заявок на тендер
• Подготовка договоров
• Замена позиции в поставке
