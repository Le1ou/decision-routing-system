# Задача для фронтенда: должности у видов работ

**Цель:** при создании/редактировании вида работ руководитель выбирает не только
допустимые **грейды**, но и допустимые **должности**; автоматическое распределение на
бэкенде уже назначает исполнителей по двум проверкам — должность сотрудника входит в
список должностей вида работ **и** его грейд входит в список грейдов.

Документ рассчитан и на человека, и на ИИ-агента: точные файлы, контракт, шаги и
критерии приёмки. **Backend менять не нужно — всё уже реализовано** (см. контракт ниже).

---

## Контекст (что уже сделано на backend)

- Новая ось матрицы допуска: таблица `type_of_work_to_post` (вид работ ↔ должности),
  симметрична существующей `type_of_work_to_grade`.
- Авто-маршрутизация (`routing_module`) подбирает исполнителя по **двум осям**:
  должность ∈ `allowedPositionIds` И грейд ∈ `allowedGradeIds`. Минимально способный,
  кулдаун, вытеснение критичными — всё как раньше, но пул кандидатов фильтруется обеими
  осями (вытеснение — тоже).
- **Пустой список должностей = ограничения по должности нет** (любая должность с
  подходящим грейдом). Это обратная совместимость: все существующие виды работ и виды
  работ, создаваемые текущим фронтом без поля, продолжают распределяться как раньше.
- **Ручное** назначение руководителем (`assignExecutor`) по-прежнему НЕ проверяет
  матрицы (решение §3.6 спецификации) — руководитель вправе назначить кого угодно.

## Контракт API (готов)

- `GET /work-types` и вложенный `workType` в карточке заявки — новое поле:
  ```jsonc
  {
    "id": "1", "name": "Починка оборудования", "departmentId": "1",
    "complexity": "easy",
    "allowedGradeIds": ["0", "1"],
    "allowedPositionIds": ["1", "4"]   // ← новое; [] = любая должность
  }
  ```
- `POST /work-types` — необязательное поле `allowedPositionIds: string[]`
  (id из `GET /positions`); не прислать или прислать `[]` = без ограничения.
- `PATCH /work-types/{id}` — `allowedPositionIds?: string[]`; список заменяется
  целиком; `[]` снимает ограничение; поле можно слать одно, без остальных.
- Справочник должностей — существующий `GET /positions`
  (`{ id, name, gradeIds }`; `gradeIds` — какие грейды бывают у должности).

## Задачи

### 1. Типы и клиент (`src/shared/api/client.ts`, `src/shared/model/domain.ts`)
- В `WorkTypeDto` / `WorkType` добавить `allowedPositionIds: string[]`.
- В `createWorkType` / `updateWorkType` payload добавить
  `allowedPositionIds?: string[]`.
- В `mapWorkTypeRecord` (`src/shared/api/mappers.ts`) — маппинг поля
  (`raw.allowedPositionIds ?? []`, по образцу `allowedGradeIds`).

### 2. Форма вида работ (`src/pages/work-types/WorkTypesPage.tsx`)
- В форму создания/редактирования добавить **мультиселект должностей** (источник —
  `positions` из `ReferenceDataProvider`), рядом с существующим выбором грейдов.
- Поле необязательное; пустой выбор показывать как «Любая должность».
- При сохранении передавать `allowedPositionIds` в create/update.

### 3. Отображение
- В списке/карточке вида работ показывать выбранные должности по именам
  (`positions.find(p => p.id === id)?.name`); для пустого списка — «Любая должность».
- ⚠️ **Заменить существующий блок** «Должности: …» на `WorkTypesPage`
  (`getAllowedPositionNames`): сейчас он вычисляет должности эвристикой
  «пересечение `positions[].gradeIds` × `allowedGradeIds`» — с введением оси
  `allowedPositionIds` это отображение может расходиться с реальным допуском
  (источник истины теперь `workType.allowedPositionIds`).
- (Опционально) в карточке заявки рядом с видом работ — те же должности из
  `application.workType.allowedPositionIds`.

### 4. Подсказка (UX, опционально)
- `positions[].gradeIds` позволяет подсветить конфликт настройки: если выбраны
  должности и грейды, у которых нет пересечения (ни у одной выбранной должности нет
  ни одного выбранного грейда), вид работ не сможет быть распределён автоматически —
  стоит показать предупреждение. Backend это не валидирует (настройка валидна, просто
  пул кандидатов будет пуст и руководитель получит уведомление-эскалацию).

## Крайние случаи

- Существующие виды работ приходят с `allowedPositionIds: []` — рендерить «Любая
  должность», не падать.
- При `PATCH` слать `allowedPositionIds` только если пользователь менял выбор
  (бэкенд заменяет список целиком).
- Должности глобальные (не по отделам) — фильтровать список не нужно.

## Критерии приёмки

- [ ] В форме вида работ можно выбрать несколько должностей; выбор сохраняется и
      возвращается после перезагрузки (`GET /work-types`).
- [ ] Пустой выбор сохраняется как `[]` и отображается «Любая должность».
- [ ] Существующие виды работ (без должностей) выглядят и работают как раньше.
- [ ] Создание вида работ без поля `allowedPositionIds` (старое поведение) — `201`.
- [ ] Backend и его контракт не менялись.

## Как проверить вручную

1. Поднять стек, войти как `orlova_m / Manager!1`.
2. На «Виды работ» создать вид работ в ИТ: грейды junior+middle, должность
   «Старший инженер».
3. Создать заявку с этим видом работ и подождать тик (~30 c): заявка должна
   назначиться **Петрову** (Старший инженер, middle), а не Иванову (Инженер, middle).
4. Убрать должности (пустой список) → следующая такая заявка может уйти любому
   свободному с подходящим грейдом.
