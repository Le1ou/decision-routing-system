# Задача для фронтенда: чат заявки

**Цель:** в карточке заявки добавить чат — переписку между автором, назначенным
исполнителем и руководителем. Backend готов; **менять его не нужно** (контракт ниже).

Документ рассчитан и на человека, и на ИИ-агента: контракт, шаги по файлам, крайние
случаи, критерии приёмки.

---

## Модель (polling, не WebSocket)

Реал-тайм имитируется опросом — у нас уже есть `usePolling` (как для списка заявок и
уведомлений). WebSocket не используется: на каждый запрос идёт Basic Auth, и текущая
прод-обвязка (Caddy) на WS не настроена. Задержки ~5 c для чата заявки достаточно.

## Контракт API (готов)

**`GET /applications/{id}/messages?afterId=<id>`** — сообщения чата + непрочитанные.
Читать может любой, кто видит карточку (автор, исполнитель, прежний исполнитель,
руководитель-в-scope); чужим — `404`. `afterId` (необязательно) — вернуть только
сообщения с `id > afterId` (инкрементальная дозагрузка для опроса).
```jsonc
{
  "items": [
    {
      "id": "12",
      "applicationId": "4",
      "authorId": "7",
      "text": "Иван, когда сможете подойти?",
      "createdAt": "2026-06-12T09:00:00Z",
      "author": {            // вложенный User (как author/executor в карточке) или null
        "id": "7", "fullName": "Фёдоров Алексей Николаевич",
        "role": "author", "departmentId": "1", "postName": "Инженер", ...
      }
    }
  ],
  "unreadCount": 2           // для ТЕКУЩЕГО пользователя (чужие сообщения после его маркера)
}
```

**`POST /applications/{id}/messages`** `{ "text": "..." }` (1..2000 символов) → `201 { "id": "13" }`.
- Писать могут **автор, назначенный исполнитель, руководитель-в-scope**.
- Непричастный (даже если технически видит) → `403`; вообще не видящий заявку → `404`.
- Заявка `completed`/`rejected` → `409` (чат read-only после закрытия; читать можно).
- Пустой/слишком длинный текст → `422`.
- Отправка автоматически сдвигает маркер прочтения отправителя (свои сообщения
  непрочитанными не считаются).

**`POST /applications/{id}/messages/read`** → `204`. Сдвигает маркер прочтения текущего
пользователя на «сейчас» (обнуляет его `unreadCount`).

**Уведомления.** При новом сообщении собеседники получают запись в общую ленту
(`GET /notifications`, текст «Новое сообщение в чате заявки …», поле `applicationId`
заполнено) — но с **анти-спамом**: пока адресат не прочитал чат, повторные сообщения
нового уведомления не создают. Фронт ничего дополнительно не делает — это обычные
уведомления, бейдж и список уже работают. (Как и все уведомления, при включённой
почте на бэкенде оно дублируется письмом на рабочий адрес — фронт не участвует.)

## Задачи

### 1. Клиент и типы (`src/shared/api/client.ts`, `src/shared/model/domain.ts`)
- Тип `ChatMessage { id, applicationId, authorId?, text, createdAt, author?: User }`.
- Методы:
  - `getMessages(creds, applicationId, params?: { afterId?: string })`
    → `{ items: ChatMessage[]; unreadCount: number }`;
  - `sendMessage(creds, applicationId, text)` → `{ id }` (POST);
  - `markChatRead(creds, applicationId)` (POST, 204).

### 2. Блок чата в карточке заявки (`src/pages/applications/ApplicationsPage.tsx`)
- Под карточкой выбранной заявки — список сообщений (автор, текст, время) и поле ввода
  с кнопкой «Отправить».
- **Право писать** определять так же, как бэкенд: текущий пользователь — автор
  (`application.authorId === currentUser.id`), назначенный исполнитель
  (`application.executorId === currentUser.id`) или руководитель-в-scope
  (`role ∈ {manager, top-manager}` и заявка его отдела / делегирована ему, либо
  top-manager). Если писать нельзя — поле ввода скрыть/задизейблить (история видна).
- Заявка `completed`/`rejected` → поле ввода скрыть (read-only), список показать.
- Ошибку POST показывать мягко (как остальные действия): `409` → «заявка закрыта»,
  `403` → «нет прав», иначе общий текст.

### 3. Опрос чата (`usePolling`)
- Для **выбранной** заявки опрашивать `getMessages(..., { afterId: <последний id> })`
  на интервале (можно чаще общего — благодаря `afterId` это дёшево) и дозаписывать
  новые сообщения в состояние.
- При открытии чата и после успешной отправки — `markChatRead`, чтобы обнулить
  непрочитанные.

### 4. Индикатор непрочитанных (опционально)
- `unreadCount` из `getMessages` показывать бейджем у заявки в списке / на иконке чата.
- (Глобально новые сообщения и так попадают в общую ленту уведомлений.)

## Крайние случаи

- Не терять черновик ввода при фоновом опросе (как уже сделано для модалок действий).
- Автоскролл вниз только если пользователь уже внизу (иначе не дёргать).
- После смены выбранной заявки чат перезагрузить с нуля (afterId сбросить).
- Сообщения приходят по возрастанию `id`; при дозагрузке добавлять в конец, не дублируя
  (фильтровать по `id`).

## Критерии приёмки

- [ ] Автор и назначенный исполнитель видят переписку и могут отправлять сообщения;
      новые появляются у собеседника в течение ≤ интервала опроса (без F5).
- [ ] Руководитель отдела (и top-manager) видит и может писать; непричастный
      пользователь чата не видит (404) и не может писать.
- [ ] В завершённой/отклонённой заявке история видна, поле ввода скрыто (POST → 409).
- [ ] Счётчик непрочитанных корректен и обнуляется после открытия/`markChatRead`.
- [ ] Backend и его контракт не менялись.

## Как проверить вручную

1. Поднять стек (`docker compose up -d`), открыть «Просмотр заявок».
2. Заявка ID 4 («Установить рабочую станцию…») уже содержит демо-чат из 4 сообщений
   между Фёдоровым (автор) и Ивановым (исполнитель).
3. Войти автором `fedorov_a / Fedorov!6` — отправить сообщение; во второй вкладке войти
   исполнителем `ivanov_i / SecretPassword!1` — увидеть его в течение интервала и
   ответить.
4. Войти `novikova_e / Novikova!5` (HR, не причастна) — чата заявки 4 быть не должно.
