# Изменения бэкенда — что знать фронтенд-команде

Реализованы пункты 1–3 плана (контракт данных, баг-фиксы, аналитика). **Фронтенд не
трогали.** Существующий контракт не сломан — все 203 интеграционных теста зелёные.
Ниже: что НОВОЕ (можно использовать), что изменилось в ПОВЕДЕНИИ, и что потребует
работы на фронте в будущем.

## 1. Новые эндпоинты — аналитика (потребуют страниц на фронте)

Готовы 4 эндпоинта (право `canViewReports`; обычный руководитель — только свой отдел,
top-manager — все; фильтры `createdFrom`/`createdTo`, отсутствие = за всё время):

- `GET /analytics/applications` — total, byStatus, byPriority, byComplexity,
  completionTimeSeconds{min,avg,max}, timeToAssignSeconds{min,avg,max}, delegations.
- `GET /analytics/executors` — по исполнителям: assigned/completed/inProgress count,
  avgHandlingTimeSeconds.
- `GET /analytics/work-types` — по видам работ: created/completed/delegated count,
  avgCompletionTimeSeconds.
- `GET /analytics/departments` — по отделам: employeeCount, applicationCount,
  completedCount, delegations{sent,received}.

> 📄 **Полный контракт полей — `docs/analytics-contract.md`** (эндпоинты, параметры,
> поля с типами и значением). Реализован полный набор §4, кроме `idleTimeSeconds` /
> `occupancyRatio` (пока `null`, помечены в контракте). По контракту можно начинать
> вёрстку страниц аналитики. Это **дополнение**: существующий `/reports` и
> `/reports/applications.xls` работают как раньше.

## 2. Изменения поведения (контракт тот же, фронт менять не нужно)

- **`/reports/applications` и `.xls` теперь scoped по отделу.** Обычный руководитель
  видит в отчёте только заявки своего отдела (раньше — все отделы, это был баг).
  top-manager — без изменений (все отделы). Фронтенд уже ограничивает выпадающий список
  исполнителей своим отделом, так что поведение стало консистентным. **Действий на фронте
  не требуется.**
- **Карточка после подтверждения/отклонения делегирования больше не падает.** Раньше
  `GET /applications/{id}` после `confirmExternalDelegation`/`declineExternalDelegation`
  возвращал 422 (баг сериализации `delegation.decidedAt`). Теперь возвращает корректный
  ISO-таймстамп. Если на фронте был воркэраунд под этот 422 — его можно убрать.
- **Подтверждённое внешнее делегирование меняет отдел заявки.** После
  `confirmExternalDelegation` у заявки `departmentId` становится целевым отделом, статус
  `new`. Карточка/списки используют `departmentId` обобщённо — действий не требуется.
- **Уведомления теперь реально создаются** при действиях: назначение исполнителя,
  делегирование в другой отдел, подтверждение/отклонение делегирования, завершение,
  отклонение руководителем. `GET /notifications` отдаёт их как обычно — фронт уже умеет.
- **Уведомления от подсистемы событий (фоновый процесс).** Бэкенд периодически
  (по умолчанию раз в 60 с) сам шлёт руководителю отдела уведомления: «истекает срок»
  (когда осталось ≤ `deadlineNotificationRatio` времени) и «заявка просрочена». Для
  фронта это обычные записи в `GET /notifications` — ничего менять не нужно, просто
  уведомлений станет больше и они будут появляться без действий пользователя.

### Требуется на фронте: редактирование настроек отдела руководителем
Добавить для руководителя отдела возможность настраивать два параметра отдела:
- **`empl_appl_delay`** (минуты) — задержка перед назначением новой заявки на сотрудника,
  который только что выполнил заявку (кулдаун). Сейчас приходит в `GET /departments` как
  `employeeApplicationDelayMinutes`.
- **`deadline_notification`** (`[0..1]`) — доля **оставшегося** времени на выполнение
  заявки, при достижении которой система шлёт уведомление **руководителю и исполнителю**
  о приближении дедлайна. Сейчас приходит в `GET /departments` как `deadlineNotificationRatio`.

> ✅ **Эндпоинт для сохранения готов:** `PATCH /departments/{id}` принимает
> `{ employeeApplicationDelayMinutes?, deadlineNotificationRatio? }` (хотя бы одно поле),
> `204` при успехе. Обычный руководитель — только свой отдел (иначе `403`),
> top-manager — любой. `deadlineNotificationRatio` валидируется в `[0..1]` (иначе `422`).
> Уведомление о приближении дедлайна теперь приходит и руководителю, и назначенному
> исполнителю, по всем открытым статусам.

### Требуется на фронте: перевод предпросмотра приоритета на плавную формулу
Итоговый приоритет теперь считается плавно: `k_времени = clamp((now − created)/(deadline −
created), 0, 1)` (без ступеней). Предпросмотр на странице «Изменение приоритетности»
(`getDeadlinePressure`, ступенчатая + захардкоженная дата «сейчас») нужно перевести на эту
же плавную формулу с реальным текущим временем, иначе предпросмотр расходится с тем, что
считает бэкенд.

### Требуется на фронте: занятость исполнителя при назначении
По решению встречи действует правило «одна заявка на исполнителя»: когда руководитель
назначает исполнителя на заявку, **его прежняя активная заявка автоматически снимается**
(возвращается в статус «Новый» с признаком «Незавершённая»). Бэкенд делает это сам.
- **Нужно на фронте:** в окне назначения исполнителя показывать, что выбранный
  исполнитель **сейчас занят другой заявкой** (предупреждение перед подтверждением).
  Признак занятости фронт может определить по списку заявок (есть ли у исполнителя
  заявка в статусе `assigned`/`inProgress`). Если удобнее получать это с бэкенда —
  скажите, добавим флаг/endpoint.

### Возможная будущая доработка (по желанию фронта)
- Признак просрочки `application.is_expired` бэкенд проставляет, но **в API карточки
  пока не отдаёт**. Если захотите бейдж «Просрочена» на заявке — скажите, добавим поле
  в ответ `GET /applications/{id}` (это +поле, не ломающее изменение).

## 3. Внутренние изменения (фронту невидимы)

- Колонка `application.priority_score` (непрерывный приоритет) — пока не наполняется и не
  отдаётся в API; появится с подсистемами приоритета/маршрутизации.
- Таблица `application_status_history` (журнал переходов) — пишется на каждое изменение
  статуса; основа аналитики.
- `priority_settings` перенесены из памяти процесса в БД — контракт `GET/PUT
  /priority-settings` не изменился, но теперь настройки переживают рестарт.
