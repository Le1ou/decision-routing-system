> Актуально на 2026-06-12. Бэкенд завершён: реализованы **все четыре подсистемы**
> (управление заявками, события, приоритет + автоматическая маршрутизация, аналитика),
> плюс релизный режим с резервным копированием, чат заявки и дублирование
> уведомлений на рабочую почту. **Фронтенд мы не трогали.**

## 1. Должности у видов работ (задача 3.6)

**Принцип назначения изменён**: авто-распределение теперь проверяет ДВЕ оси — должность
сотрудника ∈ `workType.allowedPositionIds` И грейд ∈ `workType.allowedGradeIds`.

- `GET /work-types` и вложенный `workType` карточки: поле
  `allowedPositionIds: string[]` (**пустой список = любая должность** — все
  существующие виды работ работают как раньше).
- `POST/PATCH /work-types`: необязательное `allowedPositionIds` (id из
  `GET /positions`); в PATCH список заменяется целиком, `[]` снимает ограничение.
- `GET /positions` отдаёт у каждой должности её грейды (`gradeIds`) из матрицы
  `post_grade` — пригодится для UX-подсказки: если у выбранных должностей и выбранных
  грейдов нет пересечения, авто-распределение не найдёт ни одного кандидата.
- Ручное назначение руководителем матрицы не проверяет.

Что сделать в UI: мультиселект должностей в форме вида работ, показ списка должностей
в списке/карточке («Любая должность» для пустого списка).
⚠️ Текущий блок «Должности: …» на WorkTypesPage (`getAllowedPositionNames`) вычисляет
должности **эвристикой по грейдам** — после введения оси `allowedPositionIds` он может
показывать неверный список. Заменить на данные из `workType.allowedPositionIds`.

Полное ТЗ — **`docs/frontend-task-worktype-positions.md`**.

## 2. Чат заявки (задача 3.10)

Переписка автор ↔ исполнитель ↔ руководитель в карточке заявки (polling, не WebSocket):

- `GET /applications/{id}/messages?afterId=` → `{ items, unreadCount }` (читать может
  любой, кто видит карточку; `afterId` — инкрементальная дозагрузка);
- `POST /applications/{id}/messages` `{text}` → `201` (пишут автор/исполнитель/
  руководитель-в-scope; закрытая заявка → `409`);
- `POST /applications/{id}/messages/read` → `204` (маркер прочтения).

Новое сообщение создаёт уведомление собеседнику в общую ленту (с анти-спамом) —
бейдж и список уведомлений уже работают, дополнительно ничего делать не нужно.
Заявка ID 4 в сиде содержит демо-чат для проверки.

Что сделать в UI: блок чата в карточке заявки — список сообщений + ввод, опрос через
существующий `usePolling`, право писать как у бэка, read-only для закрытых заявок.

Полное ТЗ — **`docs/frontend-task-chat.md`**.
