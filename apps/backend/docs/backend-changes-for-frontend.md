# Изменения бэкенда — что знать фронтенд-команде

> Актуально на 2026-06-11. Бэкенд завершён: реализованы **все четыре подсистемы**
> (управление заявками, события, приоритет + автоматическая маршрутизация, аналитика),
> плюс релизный режим с резервным копированием. **Фронтенд мы не трогали.**
> Существующий контракт не сломан — 260 интеграционных тестов зелёные; актуальная
> спецификация API — `docs/openapi.yaml` (перегенерирована, 26 путей).
>
> Ниже: (1) главное изменение поведения, (2) новые эндпоинты и поля,
> (3) список задач на фронте по убыванию важности.

## 1. Главное: бэкенд теперь меняет данные сам, в фоне

Раз в 30 секунд фоновый цикл:

- **пересчитывает приоритет** всех открытых заявок (фактор срока растёт со временем —
  заявка может «дозреть» до Высокого/Критичного без действий пользователя);
- **автоматически назначает** заявки в статусе «Новый» свободным подходящим
  исполнителям (по отделу, матрице грейдов, занятости и кулдауну отдела) — статус
  становится `assigned` без участия руководителя;
- **критичная заявка** распределяется немедленно при создании; если свободных нет —
  она **вытесняет** наименее приоритетную не-критичную заявку у подходящего
  исполнителя (та возвращается в «Новый» с признаком «Незавершённая»);
- помечает **просроченные** заявки и шлёт уведомления (см. ниже).

Следствия для фронта:

- ✅ **опрос по таймеру уже реализован** (`usePolling`, лёгкий список + карточка
  выбранной заявки + уведомления; интервал `VITE_POLL_INTERVAL_MS`, по умолчанию 15с) —
  фоновые изменения бэкенда доезжают до открытой страницы автоматически;
- у заявки в статусе `new` может быть заполнен `executorId`/`executor` — это след
  прежнего исполнителя после возврата/вытеснения (поле `previousExecutorId` тоже
  заполнено). Не считайте `executorId != null` признаком «назначена» — смотрите на
  `status`;
- уведомлений стало заметно больше, и они приходят без действий пользователя:
  - «Вам назначена заявка …» (авто-назначение);
  - «Заявка … снята с вас …» (вытеснение критичной / переназначение);
  - «Критичная заявка … вытеснила …», «… не удалось распределить автоматически»
    (руководителю; в т.ч. для обычных заявок без свободных исполнителей);
  - «истекает срок» / «заявка просрочена» (руководителю **и исполнителю**, по всем
    открытым статусам).

## 2. Новые эндпоинты и поля

### Аналитика — 4 эндпоинта, контракт полный
`GET /analytics/applications | /executors | /work-types | /departments`
(право `canViewReports`; обычный руководитель — свой отдел, top-manager — все;
фильтры `createdFrom`/`createdTo`, отсутствие = за всё время).

> 📄 Полный контракт полей — `docs/analytics-contract.md`. Реализован **весь** набор
> метрик, включая `idleTimeSeconds`/`occupancyRatio` (время простоя и доля занятости
> исполнителей/отделов — раньше были `null`, теперь считаются). Формат стабилен,
> страницы аналитики можно верстать; финальное согласование формата — за вами.

### Настройки отдела: `PATCH /departments/{id}`
`{ employeeApplicationDelayMinutes?, deadlineNotificationRatio? }` (хотя бы одно поле),
`204` при успехе. Обычный руководитель — только свой отдел (`403`), top-manager — любой;
`deadlineNotificationRatio` валидируется в `[0..1]` (`422`). UI для этого ещё нет — см.
задачи ниже.

### `GET /priority-settings` → read-only блок `urgent`
```jsonc
{
  "department": { ... }, "managerAuthor": { ... }, "deadline": 1.0,
  "urgent": { "thresholdHours": 24, "bonus": 0.5 }   // read-only, для предпросмотра
}
```

### Должности: `positions[].gradeIds`
`GET /positions` теперь отдаёт у каждой должности её грейды из матрицы `post_grade`
(аддитивное поле, контракт не ломает):
```jsonc
{ "items": [
  { "id": "1", "name": "Инженер",         "gradeIds": ["0", "1"] },  // junior, middle
  { "id": "3", "name": "Руководитель",    "gradeIds": ["2", "3"] }   // senior, lead
] }
```
Это закрывает пробел: у вида работ показывались только **грейды** (`allowedGradeIds`),
а какие **должности** могут к нему приступать — вычислить было нечем (матрица
должность↔грейд в API отсутствовала). Рецепт: должность подходит виду работ, если
`gradeIds ∩ workType.allowedGradeIds ≠ ∅`. См. задачу 3.6 ниже.

### Вложения: `attachments[].contentType`
В карточке заявки у вложений появился MIME-тип (например `image/png`). Маппер фронта
уже читает это поле — изображения начнут классифицироваться как «фото» автоматически,
менять ничего не нужно.

### Excel-отчёт — честный xlsx
`GET /reports/applications.xls` (путь сохранён) теперь отдаёт
`Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` и
`Content-Disposition: attachment; filename=applications.xlsx`.

### Ужесточения доступа (фронт уже ведёт себя правильно — ничего делать не нужно)
- `GET /ad/users` теперь требует `canManageEmployees` (остальным `403`). Фронт и так
  запрашивает его только при этом праве.
- Карточка чужой заявки (`GET /applications/{id}`) недоступна непричастным — `404`
  (видимость как у списка).
- Загрузка вложений — только автор / назначенный исполнитель / руководитель в scope
  (`403` остальным).
- Руководитель не может выдать сотруднику роль выше своей
  (`PATCH /employees/{id}` `role=top-manager` от обычного руководителя → `403`).
- Смена роли сотрудника через `PATCH /employees/{id}` теперь **действует сразу**
  (раньше — только после перезапуска бэкенда).

## 3. Задачи на фронте (по убыванию важности)

### 3.1. ✅ Опрос по таймеру (живое обновление) — выполнено
Реализовано по ТЗ `apps/backend/docs/frontend-task-live-updates.md`: хук `usePolling`
(гард от наложения, пауза на скрытой вкладке), опрос лёгкого списка + карточки
выбранной заявки + уведомлений, интервал `VITE_POLL_INTERVAL_MS`.

### 3.2. Пагинация списка заявок
`ApplicationsProvider` игнорирует `pagination` из `GET /applications`: запрашивается
страница по умолчанию (`pageSize=50`, максимум 100), и при бóльшем числе заявок список
в UI **молча обрезается**. Нужно либо ходить по страницам, либо явно запрашивать
`pageSize=100` и показывать «есть ещё» по `pagination.total`.

### 3.3. Предпросмотр приоритета — привести к формуле бэкенда
`calculatePriorityPreview`/`getDeadlinePressure` на странице «Изменение приоритетности»
расходятся с бэкендом в трёх местах:
1. **Захардкоженная дата** `new Date("2026-05-26…")` вместо `Date.now()`.
2. **Ступенчатый фактор срока → плавный**: бэкенд считает
   `k_времени = clamp((now − created) / (deadline − created), 0, 1) · вес_срока`.
3. **Нет бонуса срочности**: полная формула
   `score = clamp(k_отдела · k_времени + k_руководителя + бонус_срочности, 0, 1)`;
   если `(deadline − created) ≤ urgent.thresholdHours` → прибавить `urgent.bonus`
   (значения приходят в `GET /priority-settings → urgent`).

Пороги уровней: `≥0.82` critical, `≥0.62` high, `≥0.38` medium, иначе low.

### 3.4. Диапазон коэффициента отдела до 1.25
Бэкенд валидирует `priority-settings.department[...]` в **[0, 1.25]** (важные отделы
> 1 дают «Критичный» ближе к дедлайну). На странице настроек слайдер/число ограничены
`max=1` — поднять до 1.25 (managerAuthor и вес срока остаются `[0,1]`).

### 3.5. UI настроек отдела
Экран/блок для руководителя: кулдаун назначения (`employeeApplicationDelayMinutes`,
минуты) и порог уведомления о дедлайне (`deadlineNotificationRatio`, доля оставшегося
времени `[0..1]`). Текущие значения — в `GET /departments`; сохранение —
`PATCH /departments/{id}` (готов, см. §2).

### 3.6. Показ должностей у видов работ
На странице «Виды работ» (и, по желанию, в карточке заявки) показывать, какие
**должности** могут приступать к виду работ — сейчас видны только грейды. Данные уже
есть: `GET /positions → items[].gradeIds` (новое поле, см. §2); должность подходит,
если её `gradeIds` пересекаются с `workType.allowedGradeIds`. `positions` уже
загружаются в `ReferenceDataProvider` — бэкенд дорабатывать не нужно.

### 3.7. Занятость исполнителя при назначении
Правило «одна заявка на исполнителя»: при назначении занятого исполнителя его прежняя
активная заявка автоматически возвращается в «Новый» («Незавершённая»). В окне
назначения стоит предупреждать, что выбранный исполнитель занят (по списку заявок:
есть ли у него заявка `assigned`/`inProgress`). Если удобнее флаг с бэкенда — скажите.

### 3.8. Одна строка: имя файла отчёта
`ReportsPage.tsx → link.download = "applications-report.xls"` → `"applications-report.xlsx"`
(или берите имя из `Content-Disposition`). Сам файл всегда был xlsx (см. §2).

### 3.9. Страницы аналитики
По контракту `docs/analytics-contract.md` (см. §2). Формат предварительный, но полный —
если нужны изменения под вёрстку, согласуем и поправим бэкенд.

  пока не отдаёт. Скажите — добавим поле в `GET /applications/{id}` (неломающее).

## 4. Эксплуатация (фронту полезно знать)

- **Прод** разворачивается через `docker-compose.prod.yml` (Caddy, HTTPS): фронт —
  `https://<APP_DOMAIN>`, API — `https://<API_DOMAIN>`, S3 — `https://<S3_DOMAIN>`
  (домены в `.env`).
- **Релизный режим** (`SEED_ON_START=false`): база больше не пересевается демо-данными
  при рестарте; при выключении бэкенд снимает резервную копию БД в S3. В dev-режиме
  (по умолчанию) пересев на каждый старт сохраняется — демо-логины/данные те же.
- Контракт API не зависит от режима; `docs/openapi.yaml` всегда можно перегенерировать
  из живого бэкенда (`GET /openapi.json`).
