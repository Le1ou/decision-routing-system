# Задача для фронтенда: живое обновление по времени (polling)

> ✅ **ВЫПОЛНЕНО (2026-06-11).** Реализовано по этому ТЗ, включая предпочтительный
> вариант против N+1: хук `src/shared/hooks/usePolling.ts` (гард «в полёте», пауза на
> скрытой вкладке + мгновенный refresh при возврате, cleanup), интервал
> `VITE_POLL_INTERVAL_MS` → `env.pollIntervalMs`; `ApplicationsProvider` опрашивает
> **лёгкий список** (`refreshApplicationList`, слияние с кэшем карточек), полная
> карточка опрашивается только для выбранной заявки (`refreshApplicationDetail` в
> `ApplicationsPage`); уведомления опрашиваются в `AppShell`. Документ сохранён как
> описание реализации и критерии приёмки.

**Цель:** список заявок и счётчик уведомлений должны обновляться автоматически по
таймеру, без действий пользователя и без перезагрузки страницы.

Документ рассчитан и на человека, и на ИИ-агента: ниже точные файлы/функции, шаги,
крайние случаи и критерии приёмки. **Backend менять не нужно** — все эндпоинты уже есть.

---

## Контекст (как сейчас)

- **Стек:** React + react-router, состояние через провайдеры (Context). HTTP — `fetch`
  с HTTP Basic Auth на каждом запросе (заголовок формируется в `src/shared/api/client.ts`).
- **Список заявок:** `src/app/providers/ApplicationsProvider.tsx`, функция
  `refreshApplications()` тянет `GET /applications` и затем по каждой заявке
  `GET /applications/{id}` (полная карточка). Вызывается **только**: при монтировании
  (`useEffect`, завязан на `credentials`), после создания заявки (`addApplication`) и
  после действия (`performAction`). Периодического опроса нет.
- **Уведомления:** `src/widgets/app-shell/AppShell.tsx`, функция `refreshNotifications()`
  зовёт `apiClient.getNotifications(credentials)` и обновляет `notifications`/`unreadCount`.
  Вызывается **только** при монтировании и после отметки «прочитано». Периодического
  опроса нет.
- **Почему это нужно:** backend в фоне меняет данные **каждые ~30 секунд** —
  пересчитывает приоритет заявок, **автоматически назначает исполнителей** (статус
  `new` → `assigned` без действий пользователя), вытесняет заявки под критичные и
  создаёт уведомления (назначение, вытеснение, эскалация, приближение дедлайна,
  просрочка). Сейчас открытая страница этих изменений не видит до ручного refresh —
  с работающей авто-маршрутизацией это уже не «желательно», а необходимо.

## Ограничения и принципы

- **Не менять backend** и контракт API.
- Не ломать текущее поведение (обновление на монтировании и после действий должно
  остаться).
- Обновление должно быть **незаметным**: не сбрасывать пользовательский контекст
  (см. крайние случаи).
- Не создавать «шторм запросов»: одновременно не более одного активного refresh на
  ресурс; не плодить таймеры на каждый ререндер.

## Используемые эндпоинты (готовы на backend)

- `GET /applications` — лёгкий список (`id,name,status,priority,createdAt,finishedAt`).
- `GET /applications/{id}` — полная карточка (нужна только для выбранной заявки).
- `GET /notifications` — `{ items, unreadCount }`.
Авторизация — Basic Auth (как в существующих вызовах).

## Задачи

### 1. Интервал опроса в конфиге
- В `src/shared/config/env.ts` добавить `pollIntervalMs` из `VITE_POLL_INTERVAL_MS`
  (число, по умолчанию `15000`). Если переменная не задана/некорректна — использовать
  дефолт.

### 2. Переиспользуемый хук опроса (рекомендуется)
- Создать `src/shared/hooks/usePolling.ts` — хук вида
  `usePolling(callback: () => Promise<void>, intervalMs: number, enabled: boolean)`,
  который:
  - запускает `setInterval` только при `enabled === true`;
  - **гард от наложения:** не запускает новый вызов, пока предыдущий не завершился
    (флаг «в полёте» через `useRef`);
  - **пауза по видимости вкладки:** не опрашивает, когда `document.visibilityState === "hidden"`;
    при возврате на вкладку (`visibilitychange` → видима) — немедленный вызов `callback`;
  - корректно очищает таймер и слушатель в cleanup.
- (Если не делать общий хук — реализовать ту же логику инлайн в обоих местах ниже.)

### 3. Опрос списка заявок
- В `ApplicationsProvider.tsx` подключить опрос: периодически вызывать
  обновление списка, пока есть `credentials` (`enabled = Boolean(credentials)`).
- **Важно по нагрузке (N+1):** текущий `refreshApplications` грузит полную карточку по
  каждой заявке — опрашивать это каждые 15с дорого. Сделать одно из:
  - **Предпочтительно:** в опросе обновлять только лёгкий список (`GET /applications`),
    а полную карточку (`GET /applications/{id}`) грузить **только для выбранной заявки**
    (в `ApplicationsPage` по `selectedApplicationId`) — на том же интервале или при выборе.
    Для этого, возможно, разнести состояние: «список» (лёгкий) и «карточка» (полная).
  - Либо (если переделывать не хочется): оставить текущий `refreshApplications`, но
    осознанно — на маленьких объёмах данных это допустимо; интервал не делать слишком
    частым.
- Сохранить существующие вызовы обновления после создания/действия.

### 4. Опрос уведомлений
- В `AppShell.tsx` подключить опрос `refreshNotifications()` по тому же интервалу
  (`enabled = Boolean(credentials)`), не убирая обновление на монтировании и после
  «прочитано».
- Это самый ценный опрос: события (приближение дедлайна, просрочка) создаются в фоне.

### 5. Крайние случаи (обязательно учесть)
- **Сохранять выбранную заявку:** после фонового обновления списка выбранная заявка
  (`selectedApplicationId` в `ApplicationsPage.tsx`) не должна сбрасываться, если заявка
  ещё присутствует (логика удержания там уже частично есть — проверить и не сломать).
- **Не сбрасывать фильтры и сортировку** (`filters`, `sortKey`).
- **Не закрывать открытую модалку действия** (`pendingAction` в `ApplicationsPage.tsx`)
  и не терять введённые в неё данные при фоновом refresh.
- **Логаут / нет `credentials`:** опрос не идёт, состояние очищается (как сейчас).
- Ошибку запроса в фоне гасить мягко (не ронять UI, не спамить ошибками) — например,
  логировать и оставлять прежние данные до следующего успешного тика.

## Критерии приёмки (чек-лист)

- [ ] При открытой странице списка изменения в БД проявляются автоматически в течение
      ≤ `pollIntervalMs` (без действий пользователя и без F5). Проверка: создать/изменить
      заявку из другого места (вторая вкладка или через API) — строка появится/обновится;
      изменение приоритета/статуса отразится.
- [ ] Бейдж и список уведомлений обновляются в течение ≤ `pollIntervalMs`, когда backend
      создаёт уведомление в фоне (например, о приближении дедлайна/просрочке).
- [ ] Нет дублирующихся одновременных запросов одного ресурса (гард «в полёте»).
- [ ] Опрос останавливается на скрытой вкладке и сразу обновляет при возврате.
- [ ] При фоновом обновлении сохраняются: выбранная заявка, фильтры, сортировка, открытая
      модалка действия с её данными.
- [ ] Опрос не идёт без авторизации; останавливается при логауте.
- [ ] Backend и контракт API не изменены.

## Как проверить вручную

1. Поднять стек (`docker compose up -d` из корня) и войти как `orlova_m / Manager!1`.
2. Открыть «Просмотр заявок». Во второй вкладке (или через `curl`) создать заявку /
   выполнить действие — убедиться, что первая вкладка обновилась сама в пределах интервала.
3. Дождаться фонового уведомления о дедлайне (демо-заявки в mock-режиме создаются при
   старте; тик событий настраивается в `apps/backend/config.json → events.tick_seconds`) —
   убедиться, что счётчик уведомлений вырос без действий пользователя.

## Вне области задачи

- Push-обновления (WebSocket/SSE) — не требуются; достаточно опроса по таймеру. (Это
  отдельная история и потребовала бы доработки backend.)
- Любые изменения backend.
