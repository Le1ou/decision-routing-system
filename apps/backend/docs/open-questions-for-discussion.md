## Вопрос 1. Как считать `k_времени` (фактор срока) в формуле приоритета?

**Формула приоритета:** `П = k_отдела · k_времени + k_руководителя`, результат зажат в `[0,1]`.

**Суть:** `k_времени ∈ [0..1]` — насколько срочна заявка по времени. На картинке
требований (`makets/формула-расчета-приоритетности-заявки.png`) сказано «время до
завершения **относительно других заявок**», но реализованного эталона два не совпадают
по трактовке. Нужно выбрать одну формулу — **одинаковую** для фронтенда и бэкенда.

**Почему важно:** фронтенд показывает «предпросмотр приоритета» на странице настроек.
Если бэкенд считает иначе — предпросмотр будет врать пользователю.

**Варианты:**
- **A. Ступенчатая по часам до дедлайна (как сейчас во фронтенде).**
  `deadlinePressure`: >72ч → 0.35, ≤72ч → 0.65, ≤24ч → 0.9, дедлайн прошёл → 1.0;
  затем `k_времени = deadlinePressure · вес_deadline`.
  - ➕ Уже реализовано на фронте; не зависит от других заявок (дёшево и стабильно).
  - ➖ Грубые «ступени»; «относительно других заявок» из требований не учитывается.
- **B. Плавная доля прошедшего времени.**
  `k_времени = clamp((now − created) / (deadline − created), 0, 1)`.
  - ➕ Гладко растёт; интуитивно.
  - ➖ Надо поправить фронтенд, чтобы предпросмотр совпал.
- **C. Относительно других открытых заявок (буквально по картинке).**
  Нормировка остатка времени min-max по всем открытым заявкам.
  - ➖ Приоритет заявки «прыгает» от появления/закрытия чужих заявок; дорого считать.


[[Если у заявки дата дедлайна - дата создания <= n суток добавляем дополнительный коэффициент m>, где n, m - настраиваемые параметры (дефолт - 24 часа, 0.5)]]

**Пример (расхождение A vs B):** заявка создана 1 июня, дедлайн 11 июня (10 дней),
сейчас 9 июня (осталось ~48 ч). При весе `deadline = 0.5`:
- A: 48ч ≤ 72ч → 0.65 · 0.5 = **0.325**
- B: прошло 8 из 10 дней → **0.80**

При `k_отдела = 0.7`, `k_руководителя = 0`: A даёт score ≈ 0.7·0.325 = 0.23 (**low**),
B даёт 0.7·0.8 = 0.56 (**medium**). Разный уровень и разный порядок в очереди.

**Рекомендация:** A (совпадает с фронтендом, дёшево), если команда согласна, что
«относительно других заявок» — это упрощаем до пер-заявочной оценки срока.

### Живой пример из системы (почему это срочно обсудить)

Заявка №7 «Аварийный ремонт гидравлического пресса»:
- в **списке заявок** помечена как **«Критичная»** — потому что в БД `priority_id =
  critical` (так проставил сидер демо-данных; реально срочная авария);
- в **окне «Изменение приоритетности»** предпросмотр даёт **0.05 → «Низкий»**.

Расхождение вскрывает две проблемы:
1. **Источники приоритета не связаны.** Список показывает хранимый `priority_id`;
   предпросмотр считает свой `score` на клиенте. Бэкенд `priority_score` пока не
   считает вовсе. Это закроет подсистема приоритета (score → уровень), но формулу
   (вопросы 1–2) надо зафиксировать.
2. **Текущая формула почти не способна выдать «critical».** У заявки дедлайн через
   1 день (по-настоящему срочно), но:
   - фронт берёт срок = `0.35`, т.к. в `getDeadlinePressure` **захардкожено
     «сейчас» = 2026-05-26** (до дедлайна якобы > 72 ч). С реальной датой было бы `0.9`.
   - даже при `0.9`: `k_отдела 0.2 × (0.9 × k_срока 0.65) = 0.12` — далеко от порога
     `critical ≥ 0.82`.

   От порога «critical» зависит вся маршрутизация (вытеснение, запрет авто-перенаправления).
   Значит надо вместе решить: формулу `k_времени` (вопрос 1), и **калибровку
   коэффициентов/порогов**, чтобы реально срочные заявки достигали «critical».

## Вопрос 2. Как задаётся `k_руководителя` (надбавка приоритета от руководителя)?

**Суть:** в формуле есть аддитивный коэффициент руководителя. Требования
противоречивы: §5.2 и подпись формулы — «приоритет руководителя, **если он автор**»;
§5.1/§5.3 — «руководитель **может менять** приоритет заявки в статусе Новый».

**Текущее состояние фронтенда:** реализован **только** вариант «когда автор —
руководитель» (см. ниже A). UI для ручного изменения приоритета конкретной заявки **нет**.

**Варианты:**
- **A. Коэффициент отдела, авто-применяется когда автор — руководитель (как фронтенд).**
  `k_руководителя = managerAuthor[отдел_автора]`, если автор заявки имеет роль
  manager/top-manager; иначе 0. Настраивается на странице «Приоритеты».
  - ➕ Уже работает на фронте; ничего не строить.
  - ➖ Руководитель не может «вручную поднять» чужую заявку (трактовка §5.3 не покрыта).
- **B. Пер-заявочная надбавка (трактовка §5.3).**
  Новое действие `setPriorityBoost` + колонка `application.manager_priority_boost`;
  руководитель поднимает конкретную заявку в статусе `new`.
  - ➕ Полностью покрывает §5.1/§5.3.
  - ➖ **Нужен новый UI на фронте** (сейчас его нет) + действие/колонка на бэке.

**Пример влияния:** заявка с `k_отдела · k_времени = 0.5`.
- Без надбавки → score 0.50 → **medium**.
- С надбавкой `managerAuthor = 0.3` → 0.5 + 0.3 = 0.80 → **high** (почти critical).

**Рекомендация:** определиться, нужна ли ручная надбавка на заявку (§5.3) в первой
версии. Если нет — берём A (быстро, совпадает с UI). Если да — планируем B вместе с
фронтенд-командой (доработка UI).
[[Коэффициент всегда задается когда руководитель создает заявку, значение коэффициента устанавливает топ-менеджер]]

## Вопрос 3. «Одна заявка на исполнителя» при РУЧНОМ назначении руководителем

**Суть:** §6.3/§14 — на исполнителе одновременно только одна заявка
(`assigned`/`inProgress`). Авто-маршрутизация это будет соблюдать. Вопрос про **ручное**
`assignExecutor` руководителем.

**Текущее состояние:** ручное назначение **не проверяет** занятость исполнителя
(и грейд — это уже согласовано как «можно нарушать»).

**Варианты:**
- **A. Жёстко: `409`, если у выбранного исполнителя уже есть активная заявка.**
  (Кроме потока вытеснения критичной.)
  - ➕ Гарантирует правило §14.
  - ➖ **Сломает текущие интеграционные тесты** (`test_endpoints.py` назначает уже
    занятых исполнителей) — их нужно обновить. Руководителю сначала надо снять
    предыдущую заявку (`returnToNew`).
- **B. Не ограничивать ручное назначение (доверие руководителю).**
  - ➕ Гибко в форс-мажоре; тесты не трогаем.
  - ➖ Формально нарушает §14; у исполнителя может оказаться 2 активные заявки.

**Пример:** у Иванова заявка «в работе». Руководителю срочно нужно назначить ему вторую.
- A: система откажет (409) → сначала вернуть текущую в «Новый».
- B: назначит вторую → у Иванова две активные.

**Рекомендация:** A (соблюдаем §14), с оговоркой, что обновим интеграционные тесты.
[[Когда руководитель назначает человека на заявку, его прошлая заявка снимается в признак незавершенная, заявка уходит в статус новый. 
Добавить на фронт отображение при назначении исполнителя, что он сейчас занят другой заявкой]]

## Найденные баги к исправлению (не на голосование — FYI)

1. **`confirmExternalDelegation` не меняет `department_id`** — после подтверждения
   внешнего делегирования заявка остаётся в исходном отделе для маршрутизации.
   Решение: `department_id := delegated_to`.
2. **`/reports/applications` не фильтрует по отделу** — обычный руководитель видит все
   отделы. Решение: scope «только свой отдел» (как у employees/work-types).
3. **Уведомления нигде не создаются** (кроме seed) — все три подсистемы должны их
   создавать (таблица владения — в `backend-functions.md` §5).
4. **`seed.py`: `deadline_notification = 0.6–0.8`** не соответствует трактовке «доля
   оставшегося времени» — понизить до ~0.2–0.35.

## Уже принятые решения (к сведению, не обсуждаем)

- Критичность = верхний уровень приоритета (`priority='critical'`), отдельно от сложности
  работ (та — только для матрицы грейдов).
- Приоритет: непрерывный `priority_score` ∈ [0,1] (новая колонка) + 4 уровня как
  «ведро» для UI; пороги как во фронте: ≥0.82 critical / ≥0.62 high / ≥0.38 medium / low.
- Новая таблица `application_status_history` (журнал переходов) — основа аналитики.
- Подсистема событий — фоновый цикл в процессе бэкенда (asyncio в lifespan, тик ~60 с).
- Настройки приоритета — в БД, правит только top-manager (руководитель — read-only).
- Пул авто-назначения — активные исполнители своего отдела с подходящим грейдом;
  выбираем «минимально способного».
- Критичные заявки игнорируют кулдаун `empl_appl_delay`; обычные — соблюдают. - Пусть не игнорируют
- Если все подходящие заняты критичными — заявка ждёт + уведомление-эскалация
  руководителю.
- `deadline_notification` = доля **оставшегося** времени (как config 0.25).
- `is_expired` → фактор срока = 1 + уведомление руководителю.
- Отчёты/аналитика обычного руководителя — только свой отдел.
