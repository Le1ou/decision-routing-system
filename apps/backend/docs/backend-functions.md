## 1. Изменения контракта данных (нужны для подсистем 2–4)

### 1.1. Непрерывный приоритет — **[РЕШЕНО]**
Добавить `application.priority_score real` (по умолчанию 0).
- `priority_score` — непрерывное `П` из формулы (§3.2), зажатое в **[0..1]**.
  Пересчитывается подсистемой событий.
- `application.priority_id` (4 уровня) — **производное «ведро»** по порогам (§3.3);
  именно его отдаёт API (`Application.priority`) и используют правила «критичности».
- Routing и сортировка очереди работают по `priority_score`; фронтенд `priority_score`
  не потребляет — только уровень.

### 1.2. Журнал переходов статусов — **[РЕШЕНО]**
```sql
CREATE TABLE public.application_status_history (
    id              integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    application_id  integer REFERENCES public.application(application_id),
    from_status_id  integer REFERENCES public.status(status_id),  -- NULL при создании
    to_status_id    integer REFERENCES public.status(status_id),
    changed_at      timestamp with time zone NOT NULL,
    by_employee_id  integer REFERENCES public.employee(employee_id), -- NULL = система
    reason          text     -- 'manual_assign'|'auto_assign'|'critical_evict'|'delegate'|...
);
```
**Подсистема управления обязана** писать строку при КАЖДОЙ смене статуса (в
`POST /applications/{id}/actions` и при создании). На журнале строится вся аналитика
(§4) и метрики «время системы на распределение», «время на этап», «время в простое».
Без него эти метрики невозможны: `executor_at/work_at/finished_at` на `application`
перезаписываются при каждом цикле `new→assigned→new`.

### 1.3. Аддитивный коэффициент руководителя — **[РЕШЕНО, реализовано]**
`k_руководителя` — это коэффициент `managerAuthor[отдел_автора]` из настроек приоритета,
применяемый **автоматически, когда автор заявки — руководитель** (см. §3.5).
Отдельной колонки/действия не понадобилось (реализация — `priority_module.py`).

### 1.4. Настройки приоритета — персистентность — **[РЕШЕНО]**
Перенести `_priority_settings` из in-memory dict в `main.py` в БД (таблица
`priority_settings` или JSON-строка). Хранятся **три** поля (как ожидает фронтенд):
- `department` — карта `{отдел_id → коэф. k_отдела}` (default из `department.value`);
- `managerAuthor` — карта `{отдел_id → коэф. k_руководителя}`;
- `deadline` — единый вес фактора срока (0..1).

Права: **редактирует только top-manager** (`PUT /priority-settings`); обычный
руководитель видит коэффициенты своего отдела **в режиме чтения**. Это совпадает с
текущим фронтендом (`canEdit = role === "top-manager"`) и текущим бэкендом.

### 1.5. Учёт рабочего времени — **[РЕШЕНО]**
Требование §10.2 и метрики «время в работе/в простое» закрываются журналом §1.2
(суммирование интервалов статуса `inProgress`). Отдельная таблица не нужна.

## 2. Подсистема событий и исполнения решений

**Назначение:** принимает события от подсистемы управления, управляет временем,
отправляет уведомления и инициирует решения маршрутизации.

### 2.1. Способ работы — **[РЕШЕНО]**
Фоновый `asyncio`-цикл в `lifespan` FastAPI. Период тика настраиваемый
(`EVENTS_TICK_SECONDS`, по умолчанию 60 с). Работает под системным подключением
(`DBController`). На каждом тике — задачи 2.2–2.5.

### 2.2. Пересчёт приоритета
Для каждой открытой заявки (`new`, `assigned`, `inProgress`, `delegated`) пересчитать
`priority_score` (§3.2) и обновить производный `priority_id`.

### 2.3. Уведомления о приближении дедлайна — **[РЕШЕНО, реализовано]**
Для заявок во всех открытых статусах (`new`, `assigned`, `inProgress`, `delegated` —
расширено решением встречи 2026-06-09; исходно §11 требовал только `new`): вычислить
долю **оставшегося** времени `remaining = (deadline - now) / (deadline - created_at)`.
Если `remaining <= department.deadline_notification` и уведомление ещё не отправлялось —
создать уведомление руководителю отдела **и назначенному исполнителю** (§5).
- **Семантика `deadline_notification` = доля ОСТАВШЕГОСЯ времени** (как §11.2 «осталось
  25%» и `config.json` = 0.25).
- ⚠️ **Поправить seed:** текущие значения `seed.py` (0.6–0.8) под эту трактовку
  означают «уведомить почти сразу» — понизить до ~0.2–0.35.
- Защита от дублей: флаг `application.deadline_notified boolean` (добавить) либо поиск
  существующего уведомления-маркера по заявке.

### 2.4. Пометка просроченных заявок — **[РЕШЕНО, реализовано]**
Если `now > deadline` и статус не в (`completed`, `rejected`): выставить
`application.is_expired = true`, поднять фактор срока до максимума (`k_времени = 1`) и
отправить уведомление руководителю отдела и назначенному исполнителю.

### 2.5. Инициация назначения
- Заявки в `new` (в т.ч. подтверждённые из делегирования) передаются маршрутизации (§3)
  для попытки назначения с учётом настроек отдела (кулдаун, занятость).
- При появлении **критичной** заявки — внеочередной немедленный вызов маршрутизации
  (синхронно из обработчика создания/смены приоритета), не дожидаясь тика.

## 3. Подсистема принятия решений и маршрутизации

**Назначение:** автоматически распределяет и перераспределяет заявки максимально
быстро, отправляет решения в подсистему управления.

### 3.1. Определения

- **Свободный исполнитель** — **[РЕШЕНО]** сотрудник, у которого одновременно:
  `is_active = true`; `role = executor` (руководители/авторы в авто-пул **не входят**);
  отдел = `application.department_id`; грейд входит в матрицу `type_of_work_to_grade`
  для вида работ заявки; нет активной заявки в `assigned`/`inProgress`; с момента его
  `finished_at` прошло ≥ `department.empl_appl_delay` минут (кулдаун).
- **Кулдаун** — **[РЕШЕНО]** `empl_appl_delay` соблюдают **все** заявки, включая критичные
  (раннее решение «критичные игнорируют кулдаун» отменено на встрече).
- **Минимально способный исполнитель** — среди подходящих свободных — с наименьшим
  грейдом (`junior < middle < senior < lead`). При равенстве — дольше всех простаивавший.
- **Критичная заявка** — **[РЕШЕНО]** заявка с `priority_id = critical` (верхний порог
  `priority_score`, §3.3). НЕ путать с `complexity_value = critical` вида работ —
  сложность это отдельная ось только для матрицы грейдов.

### 3.2. Формула приоритета — **[РЕШЕНО на встрече]**
```
П = k_отдела · k_времени + k_руководителя + urgent_bonus
priority_score = clamp(П, 0, 1)
```
Референс базовых множителей — фронтенд (`calculatePriorityPreview`); бэкенд должен
совпасть с ним, иначе предпросмотр разойдётся с реальным приоритетом.
- `k_отдела` = `priority_settings.department[отдел_автора]` (default `department.value`).
- `k_времени` = `deadlinePressure · priority_settings.deadline`, где **[РЕШЕНО: плавная
  функция]** `deadlinePressure = clamp((now − created_at) / (deadline − created_at), 0, 1)`
  — линейно растёт от 0 (только создана) до 1 (дедлайн наступил). Кусочная/ступенчатая
  версия отменена. ⚠️ Предпросмотр во фронтенде сейчас использует ступенчатую функцию —
  его нужно перевести на эту плавную, чтобы он совпадал с бэкендом (см.
  `docs/backend-changes-for-frontend.md`).
- `k_руководителя` (см. §3.5) = `priority_settings.managerAuthor[отдел_автора]`, если
  автор заявки — руководитель/топ-менеджер; иначе 0.
- **`urgent_bonus`** — **новое**: если `(deadline − created_at) ≤ urgent_deadline_threshold_hours`,
  добавляется `urgent_deadline_bonus`. Параметры в `config.json → "priority"`
  (дефолт: порог 24 часа, бонус 0.5). Этот плоский бонус позволяет «короткосрочным»
  (срочным) заявкам дотягиваться до уровня `critical` (решает кейс заявки №7).

### 3.3. Маппинг score → уровень — **[РЕШЕНО]**
На `priority_score ∈ [0,1]` (как во фронтенде):
`≥ 0.82 → critical`; `≥ 0.62 → high`; `≥ 0.38 → medium`; иначе `low`.

### 3.4. Алгоритм распределения

**Обычная (не критичная) заявка в `new`:**
1. Найти свободных подходящих исполнителей (§3.1).
2. Есть — назначить **минимально способного**; статус → `assigned`, журнал
   `reason='auto_assign'`, уведомить исполнителя.
3. Нет — заявка остаётся в `new`, повторный разбор на следующем тике / при освобождении
   исполнителя. **Не критичные не вытесняют никого и не перераспределяются автоматически
   ни при каких условиях** (§6.4).
4. Очередь `new` обрабатывается по убыванию `priority_score`.

**Критичная заявка (в любой момент):**
1. Назначить на свободного подходящего (минимально способного).
2. Нет свободных — **вытеснение**: среди подходящих исполнителей, держащих **не
   критичную** заявку, выбрать с наименьшим `priority_score` текущей заявки; её вернуть
   в `new` (`is_unfinished=true`, `previous_executor_id`, журнал `reason='critical_evict'`);
   критичную назначить освободившемуся.
3. **Нельзя вытеснять** исполнителя с критичной текущей заявкой (§6.6).
4. **[РЕШЕНО]** Если все подходящие держат критичные — критичная заявка остаётся в `new`
   и руководителю отдела уходит **уведомление-эскалация** (разрулит вручную).

### 3.5. Коэффициент руководителя `k_руководителя` — **[РЕШЕНО на встрече]**
Надбавка применяется **всегда, когда заявку создаёт руководитель** (автор имеет роль
manager/top-manager). **Значение** надбавки задаёт **топ-менеджер** — это
per-department коэффициент `priority_settings.managerAuthor[отдел]` (редактируется через
`PUT /priority-settings`, доступно только top-manager). Отдельного действия/колонки на
заявке не нужно — это и есть текущая модель фронтенда. UI для ручной надбавки на
конкретную заявку не предусмотрен.

### 3.6. Ручное vs автоматическое назначение
- **Авто-routing** строго соблюдает матрицу грейдов и «одну заявку на исполнителя».
- **Ручное** назначение руководителем (`assignExecutor`) может **нарушать матрицу
  грейдов** (§22; фронтенд не фильтрует по грейду).
- **«Одна заявка на исполнителя» при ручном назначении** — **[РЕШЕНО на встрече, реализовано]**:
  при назначении исполнителя его прежняя **активная** заявка (`assigned`/`inProgress`)
  автоматически возвращается в статус `new` с признаком `is_unfinished=true` и
  `previous_executor_id` (link исполнителя сохраняется, чтобы он видел заявку — как в
  `returnToNew`); затем выполняется новое назначение. Реализовано в обработчике
  `assignExecutor`. **Фронтенд**: при назначении нужно показывать, что выбранный
  исполнитель сейчас занят другой заявкой (см. `docs/backend-changes-for-frontend.md`).

### 3.7. Делегирование и владение отделом — **[РЕШЕНО]**
При **подтверждении** внешнего делегирования: `application.department_id := delegated.delegated_to`
(заявка попадает в очередь нового отдела), статус `new`. При **отклонении**: заявка
возвращается в исходный отдел (`department_id` не меняется), статус `new`.
⚠️ Это исправляет текущий баг: `confirmExternalDelegation` не менял `department_id`.

## 4. Подсистема аналитики — **[РЕШЕНО: проектируем предварительно]**

**Назначение:** по команде формирует из БД статистику в виде готового к отправке JSON.
Каждая сводка — в двух режимах: за период (фильтр по датам) или за всё время (без дат).
Все длительностные метрики считаются по журналу `application_status_history` (§1.2) +
`delegated.created_at/decided_at`.

> UI для детальной аналитики пока нет. Эндпоинты и форма JSON ниже —
> **предварительные**; перед финалом согласовать с фронтенд-командой и внести в
> `openapi.yaml`.

### 4.1. Статистика заявок
Количество; распределение по приоритетам и сложностям; min/avg/max времени выполнения;
время системы на распределение/перераспределение (интервал `new → assigned` по журналу);
время «висения» без исполнителя (длительность `new`); число делегирований, принято/
отклонено; время на принятие делегирования (`decided_at - created_at`).

### 4.2. Статистика исполнителей
По каждому: число выполненных/отклонённых/делегированных/взятых в работу/назначенных/
переназначенных; min/avg/max на каждый этап; суммарное время в работе и в простое;
заявки по приоритетам/сложностям/видам работ за период.

### 4.3. Статистика видов работ
По виду работ: создано/выполнено/делегировано; приоритеты; время по отделам; время на
выполнение и в каждом статусе; кого чаще назначали.

### 4.4. Статистика отделов
Число сотрудников; время простоя (min/avg/max); доля занятости (min/avg/max); время
реагирования сотрудников и руководителя; отправлено/принято из делегирования;
статистика делегирования по отделу.

### 4.5. Эндпоинты (предварительно)
`GET /analytics/applications`, `/analytics/executors`, `/analytics/work-types`,
`/analytics/departments`; право `canViewReports`; обычный руководитель — только свой
отдел, top-manager — все; общие фильтры `from`/`to` (отсутствие = за всё время).

> **[РЕШЕНО, исправлено]** Бывший баг `/reports` (отчёт не фильтровался по отделу)
> устранён: обычный руководитель видит только свой отдел, top-manager — все;
> аналитика §4 реализована сразу с этим scope.

## 5. Владение уведомлениями (кто создаёт) — **реализовано**

Каждая подсистема создаёт уведомления для своих событий (общие хелперы —

| Событие | Создаёт | Получатель |
|---------|---------|-----------|
| Заявка назначена (авто/вручную) | Маршрутизация / Управление | Назначенный исполнитель |
| Заявка вытеснена критичной → `new` | Маршрутизация | Прежний исполнитель + руководитель отдела |
| Заявка снята при ручном переназначении («одна заявка/исполнитель») | Управление | Прежний исполнитель |
| Приближение дедлайна (все открытые статусы) | События | Руководитель отдела + назначенный исполнитель |
| Все подходящие заняты критичными (эскалация §3.4) | Маршрутизация | Руководитель отдела |
| Заявка просрочена (`is_expired`) | События | Руководитель отдела + назначенный исполнитель |
| Заявка делегирована (внеш./внутр.) | Управление | Руководитель принимающего отдела |
| Делегирование подтверждено/отклонено | Управление | Автор |
| Заявка завершена / отклонена руководителем | Управление | Автор |

## 6. Правила распределения (исходные требования — для справки)

- Приоритет заявки имеет наибольшее значение.
- Критичная заявка в любой момент пытается сразу назначиться на способного исполнителя.
- Нет свободных → критичная вытесняет наименее значимую заявку у подходящего исполнителя.
- Критичные заявки не перераспределяются автоматически, даже если новая критичная имеет
  больший приоритет.
- Не критичные распределяются по обычным правилам по мере появления свободного
  подходящего исполнителя (в порядке приоритета).
- **Не критичные заявки не перераспределяются автоматически ни при каких условиях.**
- По возможности назначать минимально способного, оставляя более ценных под критичные/
  сложные заявки.

## 7. Решения встречи (все вопросы закрыты)

1. **`k_времени`** (§3.2) — **плавная** функция
   `deadlinePressure = clamp((now − created) / (deadline − created), 0, 1)` (ступенчатая
   версия фронтенда отменена; предпросмотр фронтенда нужно перевести на плавную).
   Дополнительно введён плоский **`urgent_bonus`** (config `priority`: порог 24ч,
   бонус 0.5), чтобы срочные заявки достигали `critical`.
2. **`k_руководителя`** (§3.5) — применяется, когда автор — руководитель; значение
   (`managerAuthor[отдел]`) задаёт топ-менеджер. Отдельного действия не нужно.
3. **«Одна заявка на исполнителя»** (§3.6) — при назначении прежняя активная заявка
   исполнителя уходит в `new` (`is_unfinished`, `previous_executor_id`). **Реализовано.**
   Фронтенду: показывать занятость исполнителя при назначении.
4. **Кулдаун** (§3.1) — критичные заявки тоже соблюдают `empl_appl_delay`.

**Все четыре подсистемы (§1–§4) реализованы**, включая расчёт `priority_score`
(`priority_module.py`), подсистему событий (`events_module.py`), маршрутизацию
(`routing_module.py`, в т.ч. немедленный триггер критичной заявки при создании)
и аналитику (`analytics_module.py`, контракт — `docs/analytics-contract.md`).
