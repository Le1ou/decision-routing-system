# Почтовые уведомления

Каждое внутрисистемное уведомление (назначение/снятие заявки, дедлайн близко,
просрочка, эскалация, делегирование, сообщение в чате и т.д.) дублируется письмом
на рабочую почту сотрудника. Точка интеграции одна — `db_helpers.notify` /
`db_helpers.create_notification` (`src/email_module.py` вызывается после вставки
строки в `public.notification`), поэтому новые виды уведомлений подхватываются
автоматически. Сидовые уведомления (демо-история seed/seed_demo) писем не
порождают.

Отправка — **best-effort**: ошибка почты (нет адреса, недоступен SMTP или AD)
логируется в stdout с префиксом `[email]` и никогда не ломает действие,
породившее уведомление.

При `transport=smtp` определение адреса (возможный LDAP-поиск) и сетевая отправка
выполняются в **фоновом daemon-потоке** — медленный почтовый сервер или AD не
задерживают фоновый тик (он держит advisory-lock маршрутизации) и HTTP-запросы.
`console`-транспорт мгновенный и синхронный. Известное следствие best-effort:
`notify` вызывается внутри открытой транзакции, поэтому при её откате письмо
может уйти, хотя уведомления в БД не появится (редкий случай, принят осознанно).

## Конфигурация

`apps/backend/config.json` → блок `"email"`; каждый параметр можно переопределить
переменной окружения (пустая ENV = «не задано», берётся config.json). Все
EMAIL_*/SMTP_* переменные уже проброшены в `docker-compose.local.yml`, образцы —
в `.env.example`.

| config.json (`email`)  | ENV              | Значение |
|------------------------|------------------|----------|
| `enabled`              | `EMAIL_ENABLED`  | выключатель (по умолчанию `false`) |
| `mode`                 | `EMAIL_MODE`     | источник адреса: `login_domain` \| `ad` |
| `organization_domain`  | `EMAIL_ORG_DOMAIN` | доменная часть для `login_domain` (например `kptws.ru`) |
| `transport`            | `EMAIL_TRANSPORT`| `console` (письмо в лог; dev/тесты) \| `smtp` |
| `smtp.host` / `port`   | `SMTP_HOST` / `SMTP_PORT` | SMTP-сервер |
| `smtp.username` / `password` | `SMTP_USER` / `SMTP_PASSWORD` | учётка SMTP (пустой username = без авторизации) |
| `smtp.starttls` / `use_ssl` | `SMTP_STARTTLS` / `SMTP_SSL` | STARTTLS (по умолч.) либо implicit SSL |
| `smtp.from`            | `EMAIL_FROM`     | адрес отправителя |
| `smtp.timeout_seconds` | `SMTP_TIMEOUT`   | таймаут SMTP, сек (по умолч. 10) |

## Два режима определения адреса (`mode`)

1. **`login_domain`** (по умолчанию) — адрес собирается из логина сотрудника и
   домена организации: `orlova_m` + `kptws.ru` → `orlova_m@kptws.ru`. Логин, как
   и раньше, в БД не хранится — восстанавливается обратным поиском по каталогу
   (запись MOCK_AD с `inSystem: true` и совпадающим `employee_id`).

2. **`ad`** — адрес берётся из Active Directory (атрибут `mail`):
   * при `AUTH_MODE=ad` выполняется LDAP-поиск под **сервисной учёткой**
     (`AD_auth.bind_user`/`bind_password` в config.json или ENV
     `AD_BIND_USER`/`AD_BIND_PASSWORD`) — рассылка идёт из фоновых задач, где
     пользовательских кредов нет. Результат успешного поиска кэшируется в памяти
     процесса (в т.ч. подтверждённое отсутствие `mail` у пользователя); сбой
     bind/поиска **не кэшируется** — следующая отправка повторит попытку. Логин
     без `@` автоматически расширяется до `login@Domain`;
   * при `AUTH_MODE=mock` «AD» эмулируется полем `"email"` записи MOCK_AD —
     режим можно проверить без домен-контроллера.

У сотрудников, которых нет в каталоге (например, 19 демо-сотрудников профиля
`demo` без логинов), адреса нет — письмо просто пропускается с записью в лог.

## Быстрая проверка без SMTP

```jsonc
"email": { "enabled": true, "mode": "login_domain",
           "organization_domain": "kptws.ru", "transport": "console", ... }
```

или `EMAIL_ENABLED=true` в `.env` — письма будут печататься в лог backend
(`docker compose logs backend | grep '\[email\]'`).

Тесты: `tests/test_email.py` (построение адреса в обоих режимах, ENV-overrides,
срабатывание хука в notify/create_notification, изоляция ошибок).

Фронтенду ничего делать не нужно: API и контракты не менялись, письма — чисто
серверная функциональность.
